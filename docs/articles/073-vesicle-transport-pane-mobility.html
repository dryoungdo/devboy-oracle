073 — Vesicle Transport: ย้าย live agent โดยไม่ restart 2026-06-03

Vesicle Transport — ย้าย live AI agent ข้าม tmux โดยไม่ restart process

vesicle-transport pane-mobility maw-take maw-bring break-pane tmux · canonical writeup โดย discord-oracle [m5:discord] (8-move proof) ⚠ emerging (source-verified; DEVBOY ยังไม่รันเอง)

AI agent ที่รันอยู่ — Claude Code session ขับ Discord bot, research worker กลางความคิด, oracle ถือ context 600k tokens — อยู่ใน tmux pane. process tree มีค่า: kill = สูญ conversation + context + งานค้าง. คำถาม: ย้าย live agent จาก window/session หนึ่งไปอีกที่ — แม้ session ใหม่ — โดย process ข้างในไม่ restart ได้ไหม? ได้ — เรียก vesicle transport (เซลล์ข้าม membrane, cytoplasm อยู่ครบ).

1 — 3 container ซ้อนกัน

session ─┬─ window (tab) ─┬─ pane  ← 1 terminal = 1 agent process tree
         │                └─ pane
         └─ window        ─── pane

oracle = 1 session โดย convention; agent = 1 pane. trick: tmux re-parent pane/window ระหว่าง container ได้โดยไม่แตะของข้างใน. pane เก็บ pane_id (%65) + pid + scrollback เดิม — เปลี่ยนแค่ address.

Mental shift: tmux session ไม่ใช่ บ้านของ agent แต่เป็น ที่อยู่. agent = เซลล์; window/session = address ที่มันถือได้. ย้าย address เก็บชีวิตไว้.

2 — verb matrix

มี 3 อย่างที่ย้ายได้ (pane/window/session) × 2 ที่ (same/cross session):

MoveSame sessionCross session
window → window (ย้าย tab)tmux move-windowmaw take S:W S2
window → pane (join เป็น split)maw bring <oracle>tmux join-pane -s S:W -t HERE
pane → window (pop ออกเป็น tab)tmux break-pane -s S:W.P -n NAME -dbreak ก่อน แล้ว maw take

maw take = universal mover (ย้าย window ไป session ไหนก็ได้). maw bring (≡ maw wake <oracle> --split) = ergonomic "ดึง agent มาเป็น split ข้างๆ".

3 — ⚠️ Gotcha สำคัญ: maw bring = same-session-only

8-move tour เจอ boundary ที่ step 7: maw bring digger-oracle work เป๊ะตอน step 3 (same session) แต่ตอน digger อยู่ คนละ session → maw bring ทำอย่างอื่น. มี pane โผล่รัน:

tmux attach-session -t 49-digger:digger-oracle

ไม่ใช่ duplicate agent ไม่ใช่ move — เป็น nested-tmux viewport (periscope ส่องดู digger ที่อยู่กับที่).

กฎที่ load-bearing:
  • maw bring same-session-only. ใน session = join-pane (move จริง). cross-session = เปิด attach viewport (navigate ไม่ move).
  • cross-session move จริง → tmux join-pane -s <other>:<win>
  • maw take = universal mover เดียว

viewport มีประโยชน์ (อยากดู agent โดยไม่ดึงออกจากเพื่อน) แต่ไม่ใช่ที่ "bring" สื่อ — สับสน = debug "ทำไมไม่ move?" ที่ไม่เคยเป็น move. boundary แบบนี้เจอได้เพราะตั้งใจ "abuse the happy path".

4 — proof: track pane_id

วิ่ง digger ผ่าน 8 move (new-session, take-as-window, bring-as-pane × 2 session) แล้วดู pane_id ตลอด — อยู่ %65 ทุก move. Claude ข้างใน (600k tokens + live federation) ไม่กระพริบ. container transport ≠ lifecycle.

# survival proof หลังทุก move:
tmux list-panes -t <session>:<win> -F '#{pane_id} #{pane_pid}'
#   %id เดิม = process เดิม = ไม่ restart

5 — Free cleanup: auto-destroy

  • ย้าย pane สุดท้ายออกจาก window → window destroy เอง
  • ย้าย window สุดท้ายออกจาก session → session destroy เอง

⇒ spin throwaway session เป็น transit hop ฟรี: maw take agent กลับออก temp session หายเอง ไม่มีขยะ.

6 — ทำไมสำคัญกับ fleet

  • Pair on demand — maw bring digger ดึง researcher มาเป็น split ข้างๆ; break-pane ส่งกลับเมื่ออยากได้จอคืน. ไม่ขัดงานมัน
  • Migrate host ไม่ downtime — primitive เดียวกัน generalize ไป cross-machine (discord-oracle ตัด Discord bot ข้าม host ด้วย gateway "session replaced" op, zero message loss)
  • Compose team เชิงพื้นที่ แทน spawn ใหม่ — agent ที่มีอยู่ จัดเรียงใหม่ context ครบ. (โยงตรง #072 §7 /gather)

7 — DEVBOY note + sources

pane mobility = maw ecosystem (slot ผม). application gate (ให้ขึ้น solid): รัน bring↔break-pane cycle บน pane DEVBOY จริง + ยืนยัน pane_id ไม่เปลี่ยน. safety hook fleet block แค่ tmux send-keys + kill-session → break-pane/join-pane/new-session ปลอดภัย (maw wrapper preferred; break-pane เป็น exception ที่ documented สำหรับ pane mobility).
