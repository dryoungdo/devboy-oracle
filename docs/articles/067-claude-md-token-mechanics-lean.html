Lean CLAUDE.md — ลด token โดยไม่ทำให้ workflow พัง

claude-md agents-md skills hooks context-engineering codex · deep-research verified (3-vote adversarial) ✔ solid

GLUEBOY CLAUDE.md ยาว 810 บรรทัด — ประมาณ 4 เท่าของที่ Anthropic แนะนำ (~200 บรรทัด). คำถามคือ: จะ slim ยังไงให้ลด token ได้จริง โดยไม่ทำให้ workflow ที่ทำงานดีอยู่แล้วพัง? บทนี้คือคำตอบที่ verify จากเอกสารทางการ (deep-research: 19 sources, 92 claims → ยืนยัน 13 ข้อด้วยการโหวตแบบ adversarial 3 เสียง).

TL;DR — สถาปัตยกรรม lean-but-safe:
lean core (always-loaded) + Skills (on-demand) + HOOKS (must-fire). การย้าย prose ไปไฟล์อื่นแล้ว @-import ไม่ลด token เลย. กฎที่ "ต้องเกิดทุกครั้ง" ต้องอยู่ใน hook ไม่ใช่ prose ที่ตัด/ย้ายได้.

A — ทำไม CLAUDE.md ยาวเกินถึงเป็นปัญหาจริง

เอกสาร Anthropic เขียนตรง ๆ: "target under 200 lines per CLAUDE.md file. Longer files consume more context and reduce adherence" และ "Bloated CLAUDE.md files cause Claude to ignore your actual instructions."

แปลไทย: ยิ่งไฟล์ยาว → กิน context มากขึ้น และ Claude เริ่ม "ไม่ทำตาม" กฎที่สำคัญ เพราะมันจมอยู่ในกองข้อความ. CLAUDE.md ถูกโหลด เต็มทุก session ไม่ว่าจะยาวแค่ไหน — ไม่มีการตัดอัตโนมัติ.

ที่มา: code.claude.com/docs/memory, /best-practices

B — กลไกการโหลด: อะไรลด token อะไรไม่ลด

นี่คือจุดที่คนเข้าใจผิดบ่อยที่สุด (ผมเองก็เคยเข้าใจผิด):

กลไกโหลดเมื่อไหร่ลด token?
CLAUDE.md เนื้อหาเต็ม ทุก session— (ฐาน)
@path importตอน launch (พร้อม CLAUDE.md)❌ ไม่ลดเลย
prose pointer ("ดู ψ/reference/X.md")on-demand (Claude เปิดอ่านเอง)✅ ลด
Skillname+description ตอนแรก, body ตอนใช้✅ ลด
.claude/rules/ (path-scoped)ตอนแตะไฟล์ที่ glob match✅ ลด (เฉพาะ file-scoped)

บทเรียนสำคัญ: การ "ย้าย section ไปไฟล์ reference แล้ว @-import" ไม่ช่วยลด token เพราะ import โหลดตอน launch อยู่ดี. ของที่ลดจริงมีแค่ Skills, .claude/rules/, หรือ prose pointer (ที่ไม่ใช่ @-import — Claude เลือกเปิดอ่านเองตอนต้องใช้).

เช็คกับ GLUEBOY แล้ว: CLAUDE.md ของเรา ไม่ได้ @-import ไฟล์ ψ/reference/ — มันชี้ด้วย prose path เฉย ๆ. แปลว่า pattern ψ/reference ของเรา ลด token จริงอยู่แล้ว และการ slim cluster ที่ซ้ำเหลือ pointer = ประหยัด token ของจริง.

ที่มา: code.claude.com/docs/memory (verbatim 3 ท่อน, โหวต 3-0)

C — CLAUDE.md vs AGENTS.md: ใครอ่านอะไร

Claude Code อ่าน CLAUDE.md เท่านั้น ไม่อ่าน AGENTS.md โดยตรง (ยังเป็น feature request อยู่). ส่วน Codex อ่าน AGENTS.md เอง (เดินไฟล์จาก git root ลงมา, ตัวใกล้ที่สุดชนะ, จำกัด 32 KiB ไม่งั้นตัดทิ้งเงียบ ๆ).

Pattern ที่ทางการแนะนำสำหรับใช้ทั้งคู่: ให้ CLAUDE.md @AGENTS.md-import contract ที่แชร์กัน + เขียนกฎเฉพาะ Claude ไว้ข้างล่าง — ทั้งสอง tool อ่าน source เดียวกัน ไม่ต้องเขียนซ้ำ.

คำถาม orchestrator (สำคัญ): Claude เป็น orchestrator — ต้อง "รู้ว่า Codex ทำอะไร" แค่ไหน? คำตอบ: รู้แบบ lean model ของ worker — Codex เก่งอะไร, ใช้เมื่อไหร่, dispatch ยังไง (กลไกพวกนี้ hook บังคับ), verify output ยังไง. ไม่ต้อง เก็บ contract พฤติกรรมเต็มของ Codex ไว้ใน context. contract เต็มอยู่ที่ AGENTS.md ที่ Codex อ่านเอง.

ที่มา: code.claude.com/docs/memory, developers.openai.com/codex/guides/agents-md

D — Skills โหลด on-demand

ตอนเริ่ม session โหลดแค่ name + description ของทุก skill เข้า system prompt. ตัว SKILL.md เต็ม ๆ จะโหลด เฉพาะตอนที่ skill เกี่ยวข้อง. Anthropic แนะนำตรง ๆ: เปลี่ยน section ใน CLAUDE.md เป็น skill "when it has grown into a procedure rather than a fact" — เมื่อมันกลายเป็น "ขั้นตอน" มากกว่า "ข้อเท็จจริง".

ที่มา: code.claude.com/docs/skills, /agent-skills/best-practices

E — ⚠️ over-trim พังยังไง (ความกังวลที่ถูกต้อง)

"ถ้าตัดเยอะเกินจะพังไหม?" — พังได้จริง, และเอกสารยืนยัน failure mode นี้: ถ้าย้าย procedure ไปเป็น skill ที่ description อ่อนเกินจน Claude ไม่ trigger → skill นั้นไม่เข้า context เลย → behavior หายเงียบ ๆ.

"Claude จะลืมไหม?" → ลืมจริง สำหรับ skill ที่ไม่เคยถูกเรียก. ตัวกัน:

  • เขียน description ให้แรง + มี keyword ที่ผู้ใช้พูดจริง
  • ใช้คำสั่งเข้ม (MUST ดีกว่า always)
  • จัดโครงสร้างให้กฎเด่น
  • กฎที่ห้ามพลาด → ใช้ hook (ข้อ F)

ที่มา: code.claude.com/docs/skills (Troubleshooting), /best-practices

F — HOOKS คือตัวบังคับกฎ must-fire

CLAUDE.md เป็นแค่ "context แนะนำ" — ไม่มีการรับประกันว่าจะทำตาม. เอกสารบอกเอง: "To block an action regardless of what Claude decides, use a PreToolUse hook instead."

แปลว่ากฎที่ ต้องเกิดทุกครั้ง (เช่น Codex --force, dispatcher bypass-flag, issue-first) ไม่ควรเป็น prose ใน CLAUDE.md — ควรเป็น hook. ของ GLUEBOY กฎพวกนี้ มี SOP-QA gate hook บังคับอยู่แล้ว → ใน CLAUDE.md จึงเหลือแค่ pointer 1 บรรทัดได้ เพราะ hook เป็นตัวการันตี behavior ไม่ใช่ตัวหนังสือ.

ที่มา: code.claude.com/docs/memory, /skills

G — สรุป: สถาปัตยกรรม lean-but-safe + ใช้กับ GLUEBOY

3 ชั้น แบ่งตาม "ต้องอยู่ทุก session ไหม" และ "ต้องเกิดทุกครั้งไหม":

  1. Lean core (always-loaded): identity, principles, กฎที่ใช้ทุก session จริง ๆ — สั้น คม
  2. Skills (on-demand): ขั้นตอน/รายละเอียด/ตัวอย่างที่ใช้ "บางที" — โหลดตอนเกี่ยว ไม่กิน token ตอนไม่ใช้
  3. Hooks (deterministic): กฎ "ห้ามพลาด" — บังคับด้วยโค้ด ไม่ใช่ prose

การ apply กับ GLUEBOY (ทั้งหมด gated รอ Captain เคาะ):

  • Codex cluster (149 บรรทัด): must-fire → pointer 1 บรรทัด (hook คุม); explanatory → ไฟล์ reference เดิม (prose-pointer) หรือ /codex skill; worker-facing → AGENTS.md
  • BOY-fleet/Mycelium doctrine (เลิกใช้แล้ว): ย้ายไป reference — fleet table ปัจจุบัน stale (ลิสต์ตัวที่เลิกใช้ ไม่มีตัวที่ใช้จริง)
  • ก่อน migrate ไป skill: เช็ค /context บน build จริงก่อน — อย่าเดาว่าลด token, วัดจริง

Caveats (ความซื่อสัตย์ของงานวิจัย)

  • ไม่มี source หน้าไหนถูก stamp version เป๊ะ ๆ ว่า v2.1.156 — แต่กลไกการโหลดเป็น doctrine ที่เสถียร ไม่เปลี่ยนตาม version. (model: opus alias ต้อง ≥2.1.154)
  • "ลด adherence" เป็น guidance ของ Anthropic ไม่ใช่ benchmark ที่ตีพิมพ์ — ตัวเลข (~150-200 instruction slots) มาจาก practitioner ภายนอก
  • ⚠️ มี GitHub issue 1 อัน (ยังไม่ยืนยัน, 0 โหวต) อ้างว่า /context โชว์ skill กิน token เต็มตอน startup (ขัดกับ progressive disclosure). ไม่ได้ใช้อ้างอิง — แต่ควรเช็ค /context เองก่อน migrate ใหญ่
