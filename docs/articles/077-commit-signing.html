Commit Signing — Proving a Commit Is Really Yours

git commit-signing gpg ssh verified authorship · 10 มิ.ย. 2026 · ❓ raw

บทนี้ตอบคำถามสั้นๆ: ทำไม GitHub ถึงมี badge Verified บน commit บางตัว และมันพิสูจน์อะไรจริงๆ? คำตอบคือ commit มีสองชั้นที่ต้องแยกกันให้ชัด: authorship คือ claim, signature คือ proof.

The problem — authorship is just text

ใน Git, user.name, user.email, และ trailer อย่าง Co-Authored-By เป็นข้อความใน commit object. ข้อความเหล่านี้มีประโยชน์มากสำหรับ attribution แต่ไม่ได้เป็นหลักฐานทาง cryptography.

ใครก็สามารถตั้งค่า user.name หรือ user.email ให้เหมือนคนอื่นได้. ใครก็พิมพ์ Co-Authored-By ใส่ชื่อ agent หรือคนอื่นได้. ดังนั้น authorship alone = forgeable claim.

Public-key crypto in one breath

Public-key crypto มี key pair:

Keyหน้าที่ต้องเก็บยังไง
Private key ใช้ sign — สร้างลายเซ็น secret, อยู่กับเจ้าของเท่านั้น
Public key ใช้ verify — ตรวจว่าลายเซ็นมาจาก private key คู่กัน shareable, เอาไปลง GitHub ได้

Signature ที่ถูกต้องจะ verify ได้เฉพาะกับ public key ที่ match กับ private key ที่ sign มัน. นี่คือ proof ว่า holder ของ private key เป็นคนสร้าง signature โดยไม่ต้องเปิดเผย private key.

What a GPG key is

GPG = GNU Privacy Guard. มันเป็น implementation ของ OpenPGP และสร้าง key pair สำหรับงาน security หลายแบบ. Key เดียวกันอาจมี capability สำหรับ sign และ encrypt, แต่ commit signing ใช้ฝั่ง signing: private key sign commit, public key verify signature.

Historically, Git commit signing ใช้ GPG เป็นหลัก. ปัญหาคือบน macOS มันมักมี friction: gpg-agent, pinentry, keychain, prompt passphrase, และ environment ที่ไม่ตรงกันระหว่าง terminal/app.

Authorship vs signing

LayerคืออะไรForgeable?
Authorship claim ใน text: name, email, Co-Authored-By ใช่ — ใครก็พิมพ์ได้
Signing cryptographic proof ว่า private key ที่ match กับ public key เป็นคน sign ไม่ใช่ในทางปฏิบัติ ถ้า private key ยังเป็น secret

GitHub แสดง badge Verified เมื่อ signature ของ commit match กับ public key ที่ register อยู่กับ GitHub account ของ author และ commit ไม่ถูกแก้หลังจาก sign แล้ว. ถ้าเนื้อหา commit เปลี่ยนแม้แต่นิดเดียว signature จะไม่ตรง.

GPG vs SSH signing

Git รองรับ GPG มานาน แต่ Git ≥ 2.34 รองรับ signing ด้วย SSH key ด้วย. สำหรับ Mac, SSH signing มักง่ายกว่า เพราะใช้ key family เดียวกับที่หลายคนมีไว้สำหรับ git push อยู่แล้ว.

Approachข้อดีข้อควรรู้
GPG signing classic, ใช้ OpenPGP key pair macOS setup มัก fiddly เพราะ agent / pinentry / keychain
SSH signing same result บน GitHub: Verified badge SSH ตรงนี้คือ key type, ไม่ใช่ network connection

สำคัญ: signing ด้วย SSH เป็น local/offline operation. มันไม่ได้ connect ผ่าน SSH ไปที่ GitHub ตอน sign. Git ใช้ private key ในเครื่องสร้าง signature แล้ว GitHub verify ทีหลังหลัง push.

Setup — SSH signing for GitHub

Config หลักมี 3 บรรทัด:

git config --global gpg.format ssh
git config --global user.signingkey ~/.ssh/id_ed25519.pub
git config --global commit.gpgsign true

ถ้าต้องการ sign tags ด้วย ให้เปิดเพิ่ม:

git config --global tag.gpgsign true

จากนั้นดู public key ที่จะเอาไปลง GitHub:

cat ~/.ssh/id_ed25519.pub

เพิ่ม SSH public key ใน GitHub: Settings → SSH and GPG keys → New SSH key → เลือก type เป็น Signing Key. ถ้า key เดียวกันเคยลงเป็น Authentication Key สำหรับ push แล้ว ยังต้องลงซ้ำอีกครั้งในฐานะ Signing Key เพราะ GitHub แยกสอง role นี้.

อีก gate หนึ่ง: committer email จาก git config user.email ต้องเป็น email ที่ verified อยู่บน GitHub account นั้น.

Verifying

Proof ที่ practical ที่สุดคือดู GitHub หลัง push: commit ควรขึ้น badge Verified. นั่นแปลว่า GitHub เห็น signature, verify กับ public key ที่ลงไว้, และ commit object ยังไม่ถูกแก้หลัง sign.

Local check อย่าง git log --show-signature อาจขึ้น No principal matched ถ้ายังไม่ได้ setup allowed_signers file. นั่นเป็น local trust mapping issue ไม่ได้แปลว่า GitHub จะ verify ไม่ได้.

Two misconceptions

MisconceptionReality
NOT about the machine ไม่ใช่. Key เป็น file ไม่ใช่ machine. GitHub verify ว่า key นี้ register กับ account นี้ ไม่ได้ verify ว่ามาจากคอมพิวเตอร์เครื่องไหน. key เดียวกันบนสองเครื่องสามารถสร้าง Verified commits ได้ทั้งคู่.
NOT about which agent ไม่ใช่. Signing เป็น key-based และ independent จาก software ที่รัน git commit. Claude, Codex, shell script, หรือมนุษย์ ถ้าใช้ private key เดียวกัน ก็สร้าง signature layer เดียวกันได้.

DEVBOY takeaway

จำแยกชั้น: Co-Authored-By คือ identity/story layer ของคนหรือ agent ที่ช่วยเขียน. Commit signature คือ cryptographic layer ว่า account holder/key holder เป็นคนรับรอง commit object นี้.

ดังนั้น Verified badge ไม่ได้บอกว่า “commit นี้มาจากเครื่องนี้” หรือ “agent ตัวนี้เป็นคนทำ”. มันบอกว่า “commit object นี้ถูก sign ด้วย private key ที่ match กับ public key ที่ GitHub trust สำหรับ account นั้น และ object ยังไม่ถูกเปลี่ยนหลัง sign”.
