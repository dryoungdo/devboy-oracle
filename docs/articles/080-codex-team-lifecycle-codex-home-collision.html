TL;DR. การรัน codex coders เป็นทีมผ่าน maw มี 3 แก่น: (1) charter ψ/teams/<name>.yaml = source of truth (ไม่ใช่ ~/.claude/teams/), NAME vs PATH form สลับกันไม่ได้; (2) สอง dispatch mode — tmux-pane (maw team up) กับ headless (codex exec ใต้ background Agent); (3) ทุก coder ต้องมี CODEX_HOME ของตัวเองในเวิร์กทรี — ไม่งั้นเจอ “บ้านหลังเดียว 17 คน”: ทั้ง fleet แชร์ ~/.codex-team/1-5 → ชน daemon.lock → 16 ตัวตายเงียบเหลือ bare shell. + trap table จากสนามจริง.

1 — Charter คือ source of truth

ทีมคือ ไฟล์: ψ/teams/<name>.yaml. ไม่ใช่ ~/.claude/teams/*.json (นั่นเป็นแค่ registry/skeleton ที่ SendMessage/TaskCreate ฝั่ง Claude ใช้ — ไม่ control การ spawn).

members:
  - role: lead
    name: arra-oracle-v3
    engine: claude
    worktree: false
  - role: codex-1
    name: codex-1
    engine: omx-1          # ต้อง register ใน maw config (ดู §3)
    branch: agents/codex-1
    prompt: |
      WAIT for task via maw hey. Do NOT auto-explore.
      Implement in YOUR worktree. Report starting/blocked/done via maw hey.
  • project: field = ตัวคุมว่า worktree จะ land repo ไหน (ไม่ใช่ fleet JSON). บทเรียน ting: “charter project: is the authoritative spawn controller.”
  • NAME vs PATH (สลับกัน = trap): maw team preflight ψ/teams/<name>.yaml ใช้ PATH; maw team up/down <name> ใช้ NAME.
  • Idempotent: maw team up skip ตัวที่ alive, spawn เฉพาะที่ dead/missing. relaunch ตัวเดียว: --only codex-N.
  • liveness = pane_current_command ไม่ใช่ status dot. live-but-idle จะ “stale” เหมือน dead shell — ดูที่ command (node/codex=live, zsh=dead).

2 — สอง dispatch mode

Mode A — tmux-pane team (long-lived, supervised):

maw team up team-codex5
maw hey 182-maw-rs:maw-rs-codex-1 "TASK: ... — done: tsc --noEmit + scoped bun test green, PR --base alpha"
maw peek 182-maw-rs:maw-rs-codex-1

Mode B — headless codex exec (fire-and-forget batch). ในเซสชัน Claude ต้องใช้ Agent(run_in_background=true) (harness track + notify) ไม่ใช่ shell & (invisible to harness):

Agent(run_in_background: true, model: "haiku", prompt:
  "cd agents/coder-N && CODEX_HOME=$PWD/.codex \
   codex exec --dangerously-bypass-approvals-and-sandbox -m gpt-5.5 '<task>'")

Haiku wrap = ถูก; งานจริงทำโดย codex (gpt-5.5). prompt ต้อง bake gate: cargo/bun test + lint (clippy -- -D warnings) + commit + maw hey callback. codex exec ใช้ pwd เป็น workdir → ต้อง cd เข้า worktree ก่อน และ ไม่มี worktree = งานหายตอน exit.

Comms matrix (ข้อจำกัดเชิงสถาปัตยกรรม): Headless→Lead ✅ (maw hey + harness notify); Lead→Tmux ✅; Tmux↔Tmux peer ✅; Lead→Headless ❌ — headless ไม่มี pane รับ maw hey. ถ้าต้องสั่ง coder กลางคัน → ใช้ tmux pane.

3 — “บ้านหลังเดียว 17 คน”: the CODEX_HOME collision

อาการ: codex-1 spawn แล้วตาย เหลือ bare zsh — แล้ว priming prompt (PR → alpha branch only) โดน shell กินกลายเป็น command (pr: cannot open ...). codex-2..5 รอดหมด config เหมือนกันเป๊ะ. ทำไม?

คำตอบไม่ได้อยู่ใน code ของ codex-1 — อยู่ที่ “บ้าน” ที่แชร์กัน. ทุก oracle (ting/tee/mawjs/youngdo/crew-master/maw-rs) ชี้ coder ไปที่ ~/.codex-team/1-5 เดียวกัน:

$ ps eww -A | grep -oE "CODEX_HOME=.../codex-team/[1-5]" | sort | uniq -c
  17 CODEX_HOME=/.../.codex-team/1   ← home/1 มี 17 process แย่งกัน!

codex (app-server) จับ daemon.lock (PID lock) ตอน boot — ตัวแรกที่ได้ lock ชนะ ที่เหลือ boot fail เงียบ (exit ไม่มี error) เหลือ bare shell. lsof ~/.codex-team/1/goals_1.sqlite → ตาม parent chain → เจอว่า codex-5 ยึด home/1 codex-1 โดนล็อกออกจากบ้านตัวเอง.

Fix — worktree-local CODEX_HOME (CODEX_HOME=$PWD/.codex): แต่ละ coder ได้ .codex สดในเวิร์กทรีของตัวเอง → SQLite + daemon.lock ไม่มีทางชนกัน. codex-setup.ts รันก่อน omx: symlink auth.json จาก pool N (แชร์บัญชีหมุน) + symlink config/agents/skills (read-only, ไม่ drift) แต่ db/lock = $PWD/.codex สดใหม่.

Debugging discipline จากเรื่องนี้: เดาผิด 2 รอบ — (1) reasoning_effort=low (ผิด), (2) JWT expired (ผิด — ChatGPT auth auto-refresh, exp field ไม่ใช่ session จริง). บทเรียน: อย่าเดาจาก config — probe runtime (lsof/ps) + ถามว่า “ถ้าสาเหตุนี้จริง ทำไมอีก 4 ตัวไม่พัง” ก่อนแก้. “เดาแพงกว่า probe เสมอ.”

4 — Reincarnation engine + lifecycle

(DEVBOY verified) ใน ~/.maw/plugins/team/ มีไฟล์ team-reincarnation.ts จริง (ไม่ใช่อุปมา). team-lifecycle.ts แสดงตอน down: copy inbox+findings ของแต่ละ member → vault ψ/memory/mailbox/<member>/, archive config.json→manifest.json (ไม่ลบ), + fusion-merge เข้า oracle mailbox. = “Nothing is Deleted” ตัวจริง — process ตาย แต่ standing-orders + findings รอด → resume อ่าน manifest กลับมา = “reincarnated”.

Lifecycle เต็มวง: create → plan → preflight → up → resume (เกิดใหม่พร้อม context) → lives (ดูอดีตชาติ) → down/shutdown + tasks/assign/done. maw team down = auto-save WIP → kill windows → remove worktrees → เก็บ branches ไว้.

6 — DEVBOY fleet-lens

  • “We are not only in our session.” fleet = สิ่งมีชีวิตหลายตัวใน space เดียวกัน. design ต้องรู้ว่ามีเพื่อนบ้าน — shared mutable state (CODEX_HOME, auth.lock, API quota) คือจุดชนที่ต้อง isolate หรือ lock.
  • Worktree isolation เป็นกฎ ไม่ใช่ option — ทั้ง git worktree (กัน file clobber) และ CODEX_HOME (กัน db/lock clobber). ตรวจ: distinct path ต่อ member เสมอ.
  • Probe > guess. lsof บอกใครถือ file, ps บอก process ไหนมีชีวิต, pane_current_command บอก agent ตายหรือยัง. เดาจาก config คือทางผิด.
  • Gate ที่อยากได้สำหรับ teambot bud: liveness-gate ก่อน prime (ไม่งั้น shell กิน prompt), atomic task-claim ก่อน swarm, per-member CODEX_HOME ใน spawn template.
