A — 3 Tiers ของ Multi-Agent Work

Claude Code มี 3 ระดับของการทำงานหลาย agents พร้อมกัน — เลือกให้ถูกระดับ:

TierจำนวนToolsContextเมื่อไหร่
Subagents <3 Agent() tool แชร์ parent context งาน independent 2 ชิ้นพร้อมกัน
Team Agents 3–5 TeamCreate + SendMessage + TaskList แต่ละ agent มี context เต็มของตัวเอง งาน coordinated ที่ต้องคุยกัน
Cross-Oracle ไม่จำกัด maw talk-to / maw hey คนละ session เลย ข้ามเครื่อง, ข้าม repo

กฎเลือก Tier

2 งาน independent = Subagents (Agent() tool). งาน coordinated ที่ต้องส่ง message หากัน = Team Agents. อย่าใช้ team agents สำหรับงานง่ายๆ — overhead สูงเกินไป.

B — 6 Presets ของ /team-agents

PresetAgentsบทบาทเหมาะกับ
blank 1 1 TODO ว่าง กำหนดเอง 100%
solo 1 helper งานเดี่ยวที่ต้องการ isolation
pair 2 reviewer + writer code review, write + check
trio 3 reviewer + writer + lead งาน research + build + coordinate
review 2+ security + docs audit, compliance review
stack 4 architect + frontend + backend + tester full-stack feature build

Trio คือ default ที่ใช้บ่อยสุด — balance ดีระหว่าง coverage กับ overhead.

C — Trio Pattern (Lead + Searcher + Coder)

C.1 — บทบาท 3 ตัว

Roleหน้าที่Output
Lead (🟢) coordinate, assign, compile results, cross-verify Final report — รวมผลจากทั้งทีม
Searcher (🔵) read code, analyze, search, ส่ง findings ให้ lead Research findings + citations
Coder (🟠) write code, build, test, ส่ง output ให้ lead Working code + test results

Lead คือ bottleneck ที่ตั้งใจ — ทุก output ผ่าน lead ก่อนออก. ป้องกัน searcher กับ coder contradict กัน.

C.2 — Flow

Lead: assign tasks → Searcher + Coder
         ┌─────────────────────────────────┐
Searcher │ search → analyze → findings   │→ SendMessage → Lead
         └─────────────────────────────────┘
         ┌─────────────────────────────────┐
Coder    │ write → build → test           │→ SendMessage → Lead
         └─────────────────────────────────┘
Lead: compile → cross-verify → final report

C.3 — Cross-Agent Verification

Lead ไม่แค่รวมผล — ต้อง cross-verify: Searcher พบ pattern X, Coder implement X — ผลตรงกันหรือเปล่า? ถ้าไม่ตรง → Lead สั่ง re-check ก่อน report.

D — Lifecycle ของ Team Agents

D.1 — Prerequisites

# ต้องเปิด env var นี้ก่อน
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

ถ้าไม่เปิด → /team-agents จะไม่ทำงาน. เป็น experimental feature.

D.2 — Lifecycle Flow

TeamCreate (สร้างทีม + กำหนด preset)
  │
  ├── TaskCreate (สร้าง task, ใส่ addBlockedBy สำหรับ dependencies)
  │     ├── Task A (Searcher) — no blockers
  │     ├── Task B (Coder) — blocked by Task A
  │     └── Task C (Lead compile) — blocked by A + B
  │
  ├── Agent() spawn — แต่ละ agent เริ่มทำ task ของตัวเอง
  │
  ├── SendMessage heartbeats (ทุก 5 นาที)
  │     ├── PROGRESS — กำลังทำ
  │     ├── STUCK — ติด, ต้องการ help
  │     ├── DONE — เสร็จ
  │     └── ABORT — ล้มเหลว, หยุด
  │
  ├── Lead: compile results + cross-verify
  │
  ├── Shutdown — ปิดทีม
  │
  └── Cleanup — ลบ temp files, prune worktrees

D.3 — Agent Prompt Template

แต่ละ agent ที่ spawn ได้รับ prompt ที่มี:

  • REPO — path ของ repo ที่ทำงาน
  • TASK — task description จาก TaskCreate
  • COLOR — สี identifier (green/blue/orange) สำหรับ log
  • WORKTREE — flag ว่าใช้ worktree mode หรือไม่
  • Heartbeat protocol — PROGRESS/STUCK/DONE/ABORT ทุก 5 นาที

D.4 — Communication: SendMessage

Agents คุยกันผ่าน SendMessage — bidirectional, filesystem transport:

#Message Typeใช้เมื่อ
1TASK_ASSIGNLead มอบงาน
2PROGRESSheartbeat ทุก 5 นาที
3STUCKติดปัญหา ขอ help
4DONEเสร็จงาน
5ABORTล้มเหลว ยกเลิก
6FINDINGSSearcher ส่ง research results
7CODE_READYCoder ส่ง code ให้ review
8REVIEW_RESULTผล review กลับ
9QUESTIONถามข้ามทีม
10DIRECTIVELead สั่งเปลี่ยนทิศทาง

Transport = filesystem — durable, ไม่หายแม้ compact context.

D.5 — Worktree Mode (--worktree)

แต่ละ agent ได้ git branch แยก — ป้องกัน file conflict:

/team-agents trio --worktree

# ผลลัพธ์:
# agent-lead    → branch: team/lead
# agent-search  → branch: team/search
# agent-code    → branch: team/code

เมื่อเสร็จ → Lead merge branches กลับ. ถ้า conflict → Lead resolve.

E — Agent() vs Team-Tile เปรียบเทียบ

มิติAgent() (Subagent)Team-Tile (Team Agent)
Context window แชร์กับ parent แต่ละตัวมี context เต็มของตัวเอง
Lifetime จบเมื่อ parent turn จบ อยู่จนกว่า Shutdown หรือ task เสร็จ
Cross-repo ไม่ได้ — อยู่ใน repo เดียว ได้ผ่าน worktree + path config
Communication ผ่าน return value เท่านั้น SendMessage bidirectional, 10 types
Identity ไม่มี — anonymous worker มี role, color, prompt template
Cost ต่ำ (แชร์ context) สูง (context เต็มต่อตัว)

Rule of thumb: ถ้าไม่ต้อง SendMessage ระหว่าง agents → ใช้ Agent(). ถ้าต้อง coordinate → ใช้ Team-Tile.

F — 11 Gotchas ที่ต้องรู้

#Gotchaผลกระทบ
1 No session resume ทีมตายเมื่อ session จบ — ไม่ resume ได้
2 One team per session สร้างทีมที่ 2 ใน session เดียวไม่ได้
3 No nested teams Team agent สร้าง sub-team ไม่ได้
4 Same file = overwrites 2 agents แก้ไฟล์เดียวกัน → คนหลังทับคนแรก (ถ้าไม่ใช้ worktree)
5 Heartbeat timeout ไม่ส่ง heartbeat >10 นาที → lead อาจ mark as STUCK
6 Context per agent = full cost 3 agents = ~3x token cost ของ 1 agent
7 Env var required CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 ไม่เปิด = ไม่มีทีม
8 Worktree cleanup manual ถ้า session crash → worktrees ค้าง ต้อง git worktree prune เอง
9 No cross-session memory Team agents ไม่อ่าน ψ/ โดย default — ต้องใส่ใน prompt
10 Message ordering not guaranteed SendMessage อาจไม่เรียงตาม timestamp — ใช้ sequence number
11 Lead bottleneck Lead ไม่ respond → ทั้งทีมหยุด. Lead ต้อง responsive.
