Part A — สรุปที่ต้องรู้

Federation คืออะไร

Federation = ระบบสื่อสารข้าม Oracle ข้ามเครื่อง. GLUEBOY บน mac-studio สั่ง DEVBOY บน clinic-drdo ได้โดยตรง — ไม่ต้องผ่าน Discord หรือ manual copy.

2 Commands ที่ใช้

Commandลักษณะใช้เมื่อ
maw talk-to <oracle> "<msg>" Formal — สร้าง inbox entry ฝั่งรับ ส่ง task, directive, handoff
maw hey <oracle> "<msg>" Informal — quick ping ตรง เช็คสถานะ, ถามสั้น, verify connection

Message Flow

Oracle A → maw-js A → HMAC sign → HTTP POST → maw-js B → verify → Oracle B

ทุกข้อความถูก sign ด้วย HMAC-SHA256 ก่อนส่ง. ฝั่งรับ verify signature — ถ้าไม่ตรง reject ทันที.

Part B — Architecture

B.1 — maw-js เป็น Federation Relay

แต่ละเครื่อง (node) รัน maw-js ที่ทำหน้าที่:

  1. Local orchestrator — จัดการ Oracle sessions บนเครื่องตัวเอง
  2. Federation relay — รับ/ส่งข้อความข้ามเครื่อง
  3. HMAC gateway — sign ขาออก, verify ขาเข้า

B.2 — Message Flow Detail

1. GLUEBOY (mac-studio) รัน: maw talk-to devboy "study agent SDK"
   ↓
2. maw-js (mac-studio) รับ command
   ↓
3. Lookup namedPeers.clinic-drdo → url + key
   ↓
4. สร้าง payload:
   {
     "from": "glueboy",
     "to": "devboy",
     "body": "study agent SDK",
     "timestamp": 1748102400000,
     "nonce": "a1b2c3d4-..."
   }
   ↓
5. HMAC sign: signature = hmacSHA256(body + timestamp + nonce, federationToken)
   ↓
6. HTTP POST → https://clinic-drdo.wg:3457/federation/message
   Headers: X-Signature, X-Timestamp, X-Nonce
   ↓
7. maw-js (clinic-drdo) รับ request
   ↓
8. Verify: recompute HMAC → timingSafeEqual → check timestamp window
   ↓
9. ถ้า valid → เขียน inbox entry ให้ DEVBOY
   ↓
10. DEVBOY session ถัดไปเห็นใน ψ/inbox/glueboy/

Part C — Trust Model

C.1 — TOFU (Trust On First Use)

เหมือน SSH fingerprint — ครั้งแรกที่ peer เชื่อมต่อ จะเก็บ identity key. ครั้งต่อไปเทียบกับ key เดิม:

First connection:
  clinic-drdo → mac-studio
  mac-studio ยังไม่รู้จัก clinic-drdo
  ↓
  เก็บ key ใน namedPeers.clinic-drdo.key
  ↓
  "Trust established"

Subsequent connections:
  clinic-drdo → mac-studio
  mac-studio เทียบ key กับที่เก็บไว้
  ↓
  ตรง → accept
  ไม่ตรง → REJECT (อาจถูก impersonate)

C.2 — Security Layers

Layerป้องกันวิธี
HMAC signature tampering + impersonation HMAC-SHA256 กับ shared federation token
Timestamp window replay attack reject ข้อความเก่ากว่า 30 วินาที
Nonce duplicate messages UUID ไม่ซ้ำทุกข้อความ
Per-peer key single-peer compromise peer ตัวเดียวหลุดไม่กระทบตัวอื่น
Constant-time compare timing attack crypto.timingSafeEqual()
Network boundary external access WireGuard VPN — fleet nodes only

Part E — Configuration

E.1 — maw.config.json

// ~/.config/maw/maw.config.json
{
  "federationToken": "shared-secret-across-all-nodes",
  "federation": {
    "enabled": true,
    "port": 3457
  },
  "namedPeers": {
    "mac-studio": {
      "url": "https://mac-studio.wg:3457",
      "key": "per-peer-identity-key-mac-studio"
    },
    "homelab": {
      "url": "https://homelab.wg:3457",
      "key": "per-peer-identity-key-homelab"
    }
  }
}

chmod 600 ไฟล์นี้เสมอ — มี federation token + peer keys (ดู 049 Token Security).

E.2 — เพิ่ม Peer ใหม่

# 1. Node ใหม่ install maw-js + ตั้ง federation config
# 2. เพิ่ม peer entry ในทุก node ที่มีอยู่:
jq '.namedPeers["new-node"] = {"url":"https://new-node.wg:3457","key":"generated-key"}' \
  maw.config.json > tmp && mv tmp maw.config.json

# 3. Node ใหม่เพิ่ม existing peers ใน config ตัวเอง

# 4. Verify ทุกคู่:
maw hey new-node "ping"    # จาก existing node
maw hey clinic-drdo "ping" # จาก new node

E.3 — Troubleshooting

อาการสาเหตุแก้ไข
maw hey timeout peer offline / WireGuard down / port blocked ตรวจ wg show + curl peer-url/health
"invalid signature" federation token ไม่ตรงกัน ตรวจ federationToken ตรงทุก node
"expired" / replay rejected clock skew ระหว่าง nodes sync NTP: timedatectl set-ntp true
talk-to ส่งได้ แต่ target ไม่เห็น inbox path ผิด / oracle name mismatch ตรวจ ψ/inbox/ ของ target oracle

Part G — Security Boundary

ไม่เปิดสู่ Public Internet

Federation port (3457) bind เฉพาะ WireGuard interface — ไม่มี public exposure:

# maw-js bind เฉพาะ WireGuard IP
listen: "10.0.0.X:3457"  # WG subnet only
# ไม่ใช่ 0.0.0.0:3457   # ห้าม bind ทุก interface

Defense in Depth

Layer 1: WireGuard VPN (network isolation)
  ↓
Layer 2: HMAC signature (message authentication)
  ↓
Layer 3: Timestamp + nonce (anti-replay)
  ↓
Layer 4: Per-peer keys (blast radius control)
  ↓
Layer 5: File permissions (local token security)

attacker ต้องทะลุ ทุก layer ถึงจะ inject ข้อความปลอมเข้า fleet ได้. WireGuard อย่างเดียวก็กันได้ระดับหนึ่ง แต่ defense in depth = ปลอดภัยกว่าอาศัย layer เดียว.
