A — มันคืออะไร: protocol ไม่ใช่ knowledge

069 (/consult) ให้ “เสียงที่ปรึกษา”. 9arm-skills ตรงข้าม — ให้ วินัย (discipline-first). แต่ละ skill คือ protocol บังคับ ที่ Claude แบกไว้ทั้ง session: ทำเป็นสเต็ป, ห้ามข้าม, มีกฎที่ load-bearing ไม่ใช่คำแนะนำ. เนื้อหาอังกฤษล้วน (แม้ผู้เขียนเป็นคนไทย) — เขียนสำหรับ engineering org ที่อ่าน technical writing อังกฤษ.

4 skills, 2 หมวด: debug-mantra, post-mortem, scrutinize (engineering) + management-talk (productivity). ทุกตัวเป็น SKILL.md ล้วน ไม่มี script แนบ.

B — เจาะ 4 skills

1. debug-mantra — วินัย debug 4 ขั้น

บังคับ “ท่อง mantra” verbatim ก่อนเริ่ม debug ทุกครั้ง แล้วทำตามลำดับเป๊ะ:

  • 1. Reproducibility — ทำซ้ำได้ก่อนค่อยตั้งสมมติฐาน (flaky → ดันให้ถึง 50% determinism ก่อน)
  • 2. Fail path — debugger ก่อน → source trace + knob enumeration → in-code probe (tag [DBG-id])
  • 3. Falsify — ตั้ง 3–5 สมมติฐาน, ทดสอบ “ตัวหักล้าง” ก่อน “ตัวยืนยัน”
  • 4. Breadcrumb ledger — จด log ทุก run, ถ้า run เก่าขัดกับสมมติฐานใหม่ = สมมติฐานยังไม่ครบ
“Do not propose a fix before #1 is satisfied. A post-mortem of a hypothesis is misleading.”

2. post-mortem — บันทึก RCA ฉบับ canonical

เขียน root cause / mechanism / fix / validation หลัง fix ลงแล้ว ก่อนปิด ticket. บังคับเช็ค 4 input ก่อนเขียน (repro มี, root cause รู้, fix ระบุได้, fix ผ่าน validate). จุดเด่น: code identifier เป็น first-class — ใส่ชื่อ function/file/commit SHA ได้เต็มที่, เขียน blameless, มี worked example จริง (Tada GPU sync bug, JIRA-12345).

“Code identifiers are first-class. The whole point is that future engineers can grep their way back to the change.”

3. scrutinize — review แบบคนนอก

ลืมว่าใครเขียน อ่าน cold. 4 สเต็ป: (1) บอก intent ใน 1 ประโยค + ถามว่ามีทางที่ง่ายกว่าไหม → (2) เดินตาม code path จริง ไม่ใช่แค่ diff → (3) verify ทุก claim (edge case, concurrent, error path) → (4) report เรียงตาม severity ปิดด้วย verdict (ship / fix-then-ship / rework / reject). ห้าม “LGTM”.

“The diff is the entry point, not the scope. Bugs hide at the seams.”

4. management-talk — แปล eng → ผู้บริหาร

เขียนใหม่จากภาษา engineer ให้ leadership (VP/director/PM). กฎ: เก็บชื่อ product + JIRA key + PR, ตัด function/file/SHA/struct field, แปล mechanism เป็นเหตุ-ผลภาษาคน (race ยังเป็น race ไม่โกหก). มี 5 รูปแบบช่อง: JIRA / Slack / standup / email / meeting — แต่ละช่องมี template + worked example.

“Post-mortem owns the engineering truth; management-talk reframes it for leadership.” — 2 skill นี้ออกแบบให้ทำงานต่อกัน

C — ติดตั้ง: 2 ทาง (copy vs symlink)

ทางหลัก (แนะนำ): ผ่าน plugin marketplace

npx skills add thananon/9arm-skills

ทาง dev loop: ./scripts/link-skills.sh — น่าสนใจตรงดีไซน์:

  • สร้าง ~/.claude/skills เป็น directory จริง (ไม่ใช่ symlink เข้า repo) — มี safety check ป้องกัน $DEST เป็น symlink เข้า repo (exit พร้อม error สอนวิธีแก้)
  • แต่ละ skill เป็น symlink กลับเข้า repo → แก้ใน repo แล้ว live ทันที ไม่ต้อง reinstall
  • exclude deprecated/ in-progress/ personal/ อัตโนมัติ
ข้อควรระวัง cross-machine: วิธี symlink เหมาะ dev เครื่องเดียว. ถ้า rsync ~/.claude/skills/ ข้ามเครื่อง symlink จะพัง — ตรงกับหลัก fleet เราเป๊ะ: “copy files, not symlinks”. ข้ามเครื่องให้ใช้ npx skills add.

D — convention: plugin.json + CLAUDE.md บังคับ sync

เป็น Claude plugin จริง (มี .claude-plugin/plugin.json) แต่ minimal:

{
  "name": "9arm-skills",
  "skills": [
    "./skills/engineering/debug-mantra",
    "./skills/engineering/post-mortem",
    "./skills/engineering/scrutinize",
    "./skills/productivity/management-talk"
  ]
}

CLAUDE.md บังคับว่า skill ใหม่ทุกตัวต้องโผล่ทั้งใน README + bucket README + plugin.json; ส่วน personal/ in-progress/ deprecated/ ต้องไม่โผล่. = วินัย repo ที่ดี.

E — ⚠️ จุดเสี่ยง

  • ไม่มี LICENSE file — ไม่มีทั้งใน repo, plugin.json, README. default = all-rights-reserved → blocker สำหรับใช้ในองค์กร / fork. ต้องถามผู้เขียน
  • plugin.json ไม่มี version/description/author — marketplace listing จะโชว์แค่ path. ถ้า publish จริงควร pin version
  • ยังเด็กมาก — 3 commits, ไม่มี release/tag, maintainer คนเดียว, ไม่มี CI เช็ค sync plugin.json ↔ README
  • bucket ว่าง (misc/personal/in-progress/deprecated) — framework ยัง aspirational

F — Verdict: ควร install ไหม?

🟡 ควร install สำหรับใช้เอง/ทีม — แต่รอ LICENSE ถ้าเป็นองค์กร

คุณภาพ skill สูงจริง: prescriptive, มี worked example บน bug จริง, honest เรื่อง validation scope. สำหรับ dev เดี่ยว/ทีมเล็ก install ผ่าน npx skills add ได้เลย — ความเสี่ยง license มีผลแค่ตอนredistribute ในองค์กร. ตัวที่คุ้มที่สุด: post-mortem + scrutinize (เสริม /code-review ของเราตรงมุม “ควรมี change นี้ไหม”).

✅ ใช้ตอนไหน❌ ไม่ควรใช้ตอนไหน
ทีมที่อยากมีวัฒนธรรม post-mortem จริงจัง (RCA, blameless, code-heavy) ทีม informal “fix แล้วโพสต์ Slack จบ” — skill นี้เพิ่ม overhead
