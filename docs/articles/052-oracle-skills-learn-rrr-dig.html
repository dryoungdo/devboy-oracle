A — /learn — ส่ง AI ไปอ่านโค้ดแทนเรา

อยากเข้าใจ codebase ใหม่ ไม่ต้องอ่านเอง — สั่ง /learn แล้วให้ Haiku sub-agents หลายตัว อ่านพร้อมกัน แล้วสรุปเป็น docs ให้.

--deep5+ TESTING + API-SURFACE~10 min

A.4 — ghq + Symlink Pattern

Source code clone ผ่าน ghq (อยู่ใน ~/ghq/). ใน ψ/learn/ เก็บแค่ symlink — ไม่ duplicate source.

A.5 — Usage

/learn https://github.com/anthropics/claude-code
# clone + 3 agents อ่าน + สร้าง docs

/learn --deep maw-js
# 5 agents + full coverage

/learn --fast some-small-lib
# 1 agent, quick overview

/learn --init
# re-index existing ψ/learn/ content

หลัง learn เสร็จ → log ผ่าน arra_learn() เข้า Oracle index อัตโนมัติ.

B — /rrr — สะท้อนเพื่อเติบโต

Session retrospective — จบ session แล้วสรุปว่าทำอะไร เรียนรู้อะไร ติดตรงไหน. มี AI diary ให้ Oracle เขียนความรู้สึก + lessons learned.

B.1 — 5 Modes

CommandทำอะไรSpeed
/rrrQuick retrospective~1 min
/rrr --detailFull 13 sections~3 min
/rrr --digReconstruct จาก .jsonl logs~2 min
/rrr --deep5 parallel agents~5 min
/rrr --deep --teammate3 team agents~4 min

B.2 — 4-Step Flow

Gather (git log, diff, session context)
  ↓
Write Retrospective (ψ/memory/retrospectives/YYYY-MM/DD/HH.MM_slug.md)
  ↓
Write Lesson Learned (ψ/memory/learnings/)
  ↓
Oracle Sync (arra_learn → fleet-wide index)

B.3 — Anti-Rationalization Guard

Retrospective ที่ดีต้อง ไม่โกหกตัวเอง. /rrr มี guard ที่ scan draft หา 8 excuse patterns:

Excuse PatternGuard Response
"This was too complex"Was it complex or did you skip the hard part?
"I ran out of context"Did you spend context well?
"The API didn't work"Show the error, show what you tried
"I'll fix it next session"Concrete plan or polite abandonment?
"It's mostly done"Define "mostly" — with numbers

Guard ไม่ได้ห้ามเขียน excuse — แต่บังคับให้ ขยายความ. "Ran out of context" ผ่านได้ ถ้าบอกว่า context ถูกใช้ไปกับอะไร.

B.4 — Verification Checklist

ทุก retro ที่อ้างว่า "shipped" ต้องผ่าน:

  • มี commit hash ยืนยัน
  • AI diary มี uncomfortable truth อย่างน้อย 1 ข้อ
  • ระบุ 3+ friction points (จุดที่ติด/ช้า/ผิด)

B.5 — Oracle Root Detection

/rrr ตรวจ Oracle root ก่อนเขียน — ป้องกันเขียน retro ลง ψ/ ผิด repo.

C — /dig — ขุดประวัติ session

อยากรู้ว่าเมื่อวานทำอะไร สัปดาห์ที่แล้วทำอะไร — /dig ขุด Claude Code session history จาก .jsonl log files.

C.1 — Usage Modes

Commandทำอะไร
/digCurrent repo, 10 sessions ล่าสุด
/dig 5050 sessions ล่าสุด
/dig --allทุก session ที่มี
/dig --timelineGroup by date + day headers
/dig --deepDeep analysis พร้อม patterns

C.2 — How It Works

Resolve project dirs
  ↓
Scan .jsonl session logs
  ↓
Parse sessions:
  - sessionId
  - repoName
  - startGMT7 / endGMT7
  - durationMin
  - realHumanMessages
  - gitBranch
  - summary
  ↓
Build timeline (with gap markers)
  ↓
Output JSON / table

C.3 — Session Data Model

แต่ละ session มีข้อมูล:

FieldDescription
sessionIdUUID ของ session
repoNameRepo ที่ทำงานอยู่
startGMT7 / endGMT7เวลาเริ่ม/จบ (GMT+7)
durationMinระยะเวลา (นาที)
realHumanMessagesจำนวน messages จากคน (ไม่นับ system)
gitBranchBranch ที่ทำงานอยู่
summaryสรุปสั้นๆ ว่าทำอะไร

C.4 — Timeline Display

--timeline จัดกลุ่มตามวัน + ใส่ vibe labels:

=== 2026-05-23 (Friday) === "Infrastructure Day"

09:15  glueboy    45min  fix deploy pipeline
10:30  devboy     20min  /learn maw-rs
       . . . 45m gap . . .
11:35  glueboy    90min  doctrine sync + doctor

=== 2026-05-22 (Thursday) === "Research Day"

14:00  devboy    120min  5-agent deep trace: gale-oracle

Gap markers (. . . 45m gap) แสดงช่วงที่ไม่ได้ทำงาน — บอกได้ว่าพักเมื่อไหร่ ใช้เวลาจริงเท่าไหร่.

C.5 — Arra Integration

Discoveries จาก /dig ถูก log เข้า Arra ผ่าน arra_trace() — session history กลายเป็น searchable knowledge.
