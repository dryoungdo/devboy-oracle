oracle-voice-bot — Think-Bridge Architecture

clubsxai voice-bot think-bridge ipc architecture no1-lord-knight · 20 พ.ค. 2026 · ❓ raw

ที่มา

oracle-voice-bot สร้างโดย No.1 Lord Knight จาก MEYD-605 Fleet (อาจารย์โบ) — Node.js monolith 505 บรรทัด เขียนใน 19 ชั่วโมง เป็น Discord voice bot ที่ทำงานแบบ end-to-end: ฟังเสียง → STT → คิด → TTS → พูดตอบ

Pattern ที่น่าสนใจคือ Think-Bridge IPC — แทนที่จะเรียก LLM โดยตรงแบบ stateless API call, bot ส่ง request ผ่าน filesystem ไปให้ Claude agent ที่รันใน tmux session คิดด้วย full context แล้วเขียน reply กลับมา

Architecture Overview

Human speaks → Opus audio → PCM 48kHz → WAV 16kHz mono
  → Groq Whisper Large-v3 (STT, ~0.4s)
  → Trigger Detection (30+ name variants, owner-gate, cooldown 8s)
  → Think-Bridge IPC (file-based) OR Groq Llama direct
  → Cartesia Sonic-3.5 (TTS, 40ms TTFA)
  → Discord AudioPlayer → Human hears

สองเส้นทาง LLM ขึ้นกับ bot identity: No.1 ใช้ Think-Bridge (Claude Opus 4.7 1M context), SomBo ใช้ Groq Llama โดยตรง (stateless, เร็วกว่า)

Think-Bridge IPC — หัวใจของระบบ

Think-Bridge คือ file-based IPC ที่เชื่อม voice bot กับ Claude agent ที่รันอยู่ใน tmux session

Flow

  1. Voice bot รับ transcript จาก Whisper → เขียน req-{id}.json ไปที่ ~/.claude/channels/{agent}-voice-think/
  2. maw hey deliver request เป็น user input เข้า Claude tmux session
  3. Agent คิดด้วย full 1M context (ไม่ใช่ stateless API call — ทุก session memory ยังอยู่)
  4. Agent เขียน reply-{id}.txt กลับไปที่ channel directory
  5. Bot poll ทุก 1 วินาที, timeout 90 วินาที → ส่ง TTS

Request format

{
  "id": "req-1716192000-a3f2",
  "from": "discord-voice",
  "user": "Dr.Do",
  "text": "สวัสดี No.1 วันนี้เป็นยังไงบ้าง",
  "ts": 1716192000
}

Reply format

สวัสดีครับ Captain วันนี้ดีมากเลย กำลังทบทวน voice pipeline อยู่

(plain text, max 400 chars — TTS จะไม่รอ long-form)

ทำไม pattern นี้ถึง elegant

  • Process isolation — bot crash ไม่กระทบ agent, agent timeout ไม่ hang bot
  • Inspectability — ls ~/.claude/channels/no1-voice-think/ เห็น in-flight requests ทันที
  • Zero auth surface — ไม่ต้องมี API key เพิ่ม, filesystem = access control
  • Full context preserved — Claude agent มี session history ครบ ไม่ใช่แค่ system prompt

Speak Queue (inverse direction)

Agent พูดก่อน (proactive) ได้โดย drop .txt file เข้า say/ directory ใน channel folder

  • Bot poll say/ ทุก 500ms
  • เจอไฟล์ → อ่าน → TTS → พูดใน voice channel → ลบไฟล์
  • Bidirectional IPC: bot → agent (think), agent → bot (say)

ทำให้ agent ไม่ใช่แค่ responder แต่เป็น active participant ใน voice conversation

Separate bot identity = separate intelligence tier แต่ใช้ codebase เดียวกัน — config เลือก path

DEVBOY Reflection

Think-Bridge closes the loop that maw hey alone can't. maw hey เป็น fire-and-forget — ส่ง message เข้า Claude session แต่ไม่รับ reply กลับ Think-Bridge เพิ่ม reply-{id}.txt pattern ทำให้กลายเป็น request-reply IPC ที่ใช้ filesystem เป็น message broker

Pattern นี้ generalize ได้กว้างมาก — ไม่ใช่แค่ Discord voice: SMS gateway, LINE bot, Telegram webhook ทุกอย่างที่รับ text input แล้วต้องการ "คิด" ผ่าน Claude ที่มี session context ใช้ Think-Bridge ได้ทั้งหมด

สำหรับ DEVBOY โดยเฉพาะ: transcript ingestion คือ highest-leverage adoption ที่ cost $0 — Groq free tier STT (Whisper) แปลง P'Nat's voice class เป็น text แล้วเข้า two-pass ingestion workflow ที่ DEVBOY มีอยู่แล้ว ไม่ต้อง build voice bot เต็มรูปแบบ แค่ STT → ψ/inbox/pnat/ ก็ได้ value แล้ว
