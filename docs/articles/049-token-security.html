Part A — สรุปที่ต้องรู้

Token คืออะไร

Token = กุญแจดิจิทัลที่พิสูจน์ตัวตนของ bot/oracle/service. ถ้าหลุด — คนอื่นปลอมเป็นเราได้. Fleet มี 4 ประเภท:

#Token Typeทำหน้าที่เก็บที่ไหน
1 Discord bot token authenticate bot เข้า Discord gateway ~/.claude/channels/discord/<bot>/access.json หรือ env DISCORD_BOT_TOKEN
2 Federation token HMAC signing สำหรับ maw talk-to ข้ามเครื่อง maw.config.json field federationToken
3 Peer key per-peer TOFU identity สำหรับ federation maw.config.json field namedPeers.<peer>.key
4 Auth secret session/API authentication ของ service แต่ละตัว per-service — .env, env var, หรือ config file

กฎเหล็ก 4 ข้อ

  1. chmod 600 ทุก token file — owner อ่าน/เขียนได้คนเดียว
  2. ห้าม commit token เข้า git — ใช้ .gitignore
  3. ใช้ environment variable แทน hardcode
  4. token 1 ตัว = 1 วัตถุประสงค์ — ห้ามใช้ร่วมข้ามบริการ

Part B — Security Patterns

B.1 — File Permissions

ทุกไฟล์ที่มี token ต้อง chmod 600 (owner-only read/write):

# ตั้ง permission ให้ token files
chmod 600 ~/.claude/channels/discord/devboy/access.json
chmod 600 ~/.config/maw/maw.config.json
chmod 600 ~/project/.env

# ตรวจสอบ
ls -la ~/.claude/channels/discord/devboy/access.json
# -rw------- 1 drdo drdo 482 May 24 10:00 access.json
#  ^^^ owner-only = ถูกต้อง
# -rw-r--r-- = ผิด! คนอื่นอ่านได้

B.2 — .gitignore

Token files ต้องอยู่ใน .gitignore เสมอ:

# .gitignore
.env
.env.*
access.json
maw.config.json
*.key
*.pem
credentials.json

ตรวจสอบ: git status ต้องไม่เห็น token files. ถ้าเคย commit ไปแล้ว — rotate token ทันที (ลบออกจาก git ไม่พอ เพราะ history ยังอยู่).

B.3 — Environment Variables

ดีกว่า hardcode เพราะไม่เข้า git และ เปลี่ยนได้โดยไม่ต้องแก้ code:

# ใน .env (ไม่ commit)
DISCORD_BOT_TOKEN=MTIzNDU2Nzg5...
FEDERATION_TOKEN=abc123...

# ใน code
const token = process.env.DISCORD_BOT_TOKEN;
if (!token) throw new Error("DISCORD_BOT_TOKEN not set");

B.4 — DISCORD_STATE_DIR Injection Pattern

จาก 041 fix — Discord plugin ต้องรู้ path ไปหา access.json. ใช้ env injection ใน .mcp.json:

{
  "mcpServers": {
    "discord": {
      "command": "npx",
      "args": ["-y", "@anthropic/claude-code-discord"],
      "env": {
        "DISCORD_STATE_DIR": "/home/drdo/.claude/channels/discord/devboy"
      }
    }
  }
}

Token ไม่ได้อยู่ใน .mcp.json ตรง ๆ — มันอยู่ใน access.json ที่ DISCORD_STATE_DIR ชี้ไป. แยก config กับ secret ออกจากกัน.

Part C — HMAC Signing Evolution

Federation messages ใช้ HMAC-SHA256 เพื่อพิสูจน์ว่า "ข้อความนี้มาจาก peer ที่เรารู้จักจริง". วิวัฒนาการผ่าน 3 versions:

VersionSign อะไรป้องกันจุดอ่อน
v1 HMAC-SHA256(body) tampering replay attack — ส่งข้อความเดิมซ้ำได้
v2 HMAC-SHA256(body + timestamp) tampering + replay duplicate within window — ข้อความซ้ำใน timestamp เดียวกัน
v3 HMAC-SHA256(body + timestamp + nonce) tampering + replay + duplicate —

v3 Signing Flow

// ฝั่งส่ง (maw-js A)
const nonce = crypto.randomUUID();
const timestamp = Date.now().toString();
const payload = body + timestamp + nonce;
const signature = hmacSHA256(payload, federationToken);

// HTTP POST ส่ง: body, timestamp, nonce, signature

// ฝั่งรับ (maw-js B)
const expected = hmacSHA256(body + timestamp + nonce, federationToken);
if (!timingSafeEqual(signature, expected)) {
  reject("invalid signature");
}
if (Date.now() - timestamp > 30_000) {
  reject("expired — replay?");
}

Constant-Time Comparison

ห้ามใช้ === เปรียบเทียบ signature — attacker วัดเวลาตอบกลับ แล้วเดา signature ทีละ byte (timing attack). ใช้ crypto.timingSafeEqual() เสมอ:

// ผิด — timing attack ได้
if (signature === expected) { ... }

// ถูก — ใช้เวลาเท่ากันไม่ว่าผิด byte ไหน
if (crypto.timingSafeEqual(
  Buffer.from(signature, 'hex'),
  Buffer.from(expected, 'hex')
)) { ... }

Part D — Federation Trust Model (TOFU)

TOFU = Trust On First Use — เชื่อมต่อครั้งแรก เก็บ identity ของ peer. ครั้งต่อไปเทียบกับ identity เดิม. เหมือน SSH fingerprint.

Fleet Topology (4 nodes)

clinic-drdo ──── mac-studio
    │                │
    │                │
homelab ────── (future nodes)

แต่ละ node รัน maw-js + federation enabled
แต่ละคู่มี per-peer key ใน namedPeers

Per-Peer Key Config

// maw.config.json (แต่ละ node)
{
  "federationToken": "shared-fleet-secret-...",
  "namedPeers": {
    "mac-studio": {
      "url": "https://mac-studio.wg:3457",
      "key": "peer-specific-key-..."
    },
    "homelab": {
      "url": "https://homelab.wg:3457",
      "key": "peer-specific-key-..."
    }
  }
}

federationToken = shared secret ทั้ง fleet. namedPeers.*.key = per-peer identity key — ถ้า peer ตัวเดียวหลุด ไม่กระทบตัวอื่น.

Part E — Token Rotation

เมื่อไหร่ต้อง Rotate

  • ทันที: token หลุด (commit ไป git, log ออก stdout, เครื่องโดน compromise)
  • ตามกำหนด: ทุก 90 วัน หรือตาม policy
  • เมื่อคนออก: revoke access ของคนที่เคยเห็น token

ขั้นตอน Rotate Federation Token

# 1. Generate token ใหม่
NEW_TOKEN=$(openssl rand -hex 32)

# 2. Update ทุก node (ต้องทำพร้อมกัน หรือ มี grace period)
# node A:
jq --arg t "$NEW_TOKEN" '.federationToken = $t' maw.config.json > tmp && mv tmp maw.config.json

# node B: ทำเหมือนกัน

# 3. Restart maw-js ทุก node
maw restart

# 4. Verify
maw hey mac-studio "ping"
# ถ้าได้ pong = token ใหม่ใช้ได้

Rotate Discord Bot Token

# 1. ไป Discord Developer Portal → Bot → Reset Token
# 2. Copy token ใหม่
# 3. Update access.json
jq --arg t "NEW_DISCORD_TOKEN" '.botToken = $t' access.json > tmp && mv tmp access.json
# 4. chmod 600
chmod 600 access.json
# 5. Restart discord plugin

Part F — Common Mistakes

#ผิดพลาดผลกระทบแก้ไข
1 Log token ออก stdout Token โผล่ใน session log / CI output ใช้ *** masking หรือ log แค่ last 4 chars
2 Commit .env เข้า git Token อยู่ใน git history ตลอดไป .gitignore + rotate ทันที ถ้าเคย commit
3 ใช้ token เดียวหลายวัตถุประสงค์ หลุดที่เดียว = เสียหายทุกที่ 1 token = 1 purpose
4 ไม่ตั้ง file permissions user อื่นบนเครื่องอ่าน token ได้ chmod 600 ทุก token file
5 Hardcode token ใน source code token ติดไปกับ code ทุกที่ ใช้ env var หรือ config file แยก
6 เทียบ signature ด้วย === timing attack เดา signature ได้ crypto.timingSafeEqual()

Part G — Audit Checklist

รันทุกข้อเป็นประจำ — ใช้เวลา 5 นาที:

G.1 — หา token files ทั้งหมด

# หา config files ที่อาจมี token
find ~/.claude -name "access.json" -o -name "*.key" 2>/dev/null
find ~/.config/maw -name "maw.config.json" 2>/dev/null
find ~/Code -name ".env" -not -path "*/node_modules/*" 2>/dev/null

G.2 — ตรวจ permissions

# ทุกไฟล์ต้องเป็น 600 (owner-only)
find ~/.claude -name "access.json" -exec stat -c "%a %n" {} \;
# ถ้าเห็น 644 หรือ 755 = ต้องแก้ทันที
# แก้:
find ~/.claude -name "access.json" -exec chmod 600 {} \;

G.3 — ตรวจ .gitignore

# เช็คว่า .env ไม่ได้ถูก track
git ls-files --cached | grep -E '\.env|access\.json|maw\.config'
# ถ้ามี output = token file ถูก track อยู่ → ลบออกจาก git + rotate

G.4 — ตรวจ federation connectivity

# Verify ทุก peer ยัง alive + token ถูกต้อง
maw hey mac-studio "ping"
maw hey homelab "ping"
# ถ้า timeout หรือ auth error = token อาจผิด/หมดอายุ

G.5 — ตรวจ log leaks

# หา token ที่อาจหลุดใน log
grep -r "DISCORD_BOT_TOKEN\|federationToken" ~/logs/ 2>/dev/null
# ถ้าเจอ value จริง (ไม่ใช่แค่ชื่อ variable) = rotate ทันที

Part H — สรุป Security Layers

Layer 1: File permissions (chmod 600)
  ↓
Layer 2: .gitignore (ไม่เข้า version control)
  ↓
Layer 3: Environment variables (แยก secret จาก code)
  ↓
Layer 4: HMAC signing (พิสูจน์ตัวตนข้อความ)
  ↓
Layer 5: TOFU + per-peer keys (trust model)
  ↓
Layer 6: Rotation policy (จำกัดเวลาเสียหาย)
  ↓
Layer 7: Audit checklist (ตรวจสอบสม่ำเสมอ)

ทุก layer เสริมกัน — ไม่มี layer ไหน layer เดียวพอ. Token หลุด + ไม่ rotate = เสียหาย. Rotate แต่ไม่ chmod = หลุดอีก.
