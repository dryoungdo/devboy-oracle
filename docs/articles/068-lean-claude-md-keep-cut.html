A — ⚠️ `.claude/rules/` ไม่ฟิตกับ behavioral doctrine

.claude/rules/ มี frontmatter key เดียว = paths (glob). มี 2 โหมด:

  • ไม่มี `paths` → โหลด ทุก session priority เท่า CLAUDE.md (= cost เท่ากัน ไม่ประหยัด token เลย แค่แยกไฟล์)
  • มี `paths` → โหลด เฉพาะตอน Claude อ่านไฟล์ที่ตรง glob — ไม่ใช่ตอน Write/Edit, ไม่ใช่ behavioral trigger

⇒ doctrine เรื่อง delegation / codex-dispatch / identity มัน ไม่ผูกกับการเปิดไฟล์ → ไม่มี read event มา fire → มัน หายเงียบ ๆ. สรุป: .claude/rules/ เหมาะกับกฎ เฉพาะไฟล์ (เช่น "ไฟล์ใน src/api ต้อง validate input") ไม่ใช่กฎ พฤติกรรม. behavioral doctrine ต้องอยู่ใน CLAUDE.md ที่ lean หรือ Skill (+ hook ถ้า must-fire).

เสริม: ไม่ทำให้ PreToolUse hook พัง (hook fire ตาม tool-name อิสระจาก prose). risk หลัก = version-fragility (หลาย bug ข้าม version) → pin + test ด้วย /context บน build จริงก่อน.

ที่มา: code.claude.com/docs/memory + claude-code issues 23478/38487/16299/21858/17204

B — กฎ keep/cut: per-line necessity test

หลักทางการสำหรับ "เก็บอะไร":

Per-line test: "ถ้าลบบรรทัดนี้ Claude จะทำพลาดไหม? ถ้าไม่ → ตัด"
  • เก็บ: คำสั่งที่เดาไม่ได้, style ที่ต่างจาก default, architectural decision, env quirk, gotcha ที่ไม่ชัด, กฎ "always do X" ที่ใช้กว้าง
  • ย้ายออก (skills/rules): ขั้นตอนหลายสเต็ป, ความรู้ที่ใช้บางที, workflow เฉพาะพื้นที่
  • ตัดทิ้ง: อะไรที่ Claude เดาจากโค้ดได้, convention มาตรฐานที่มันรู้แล้ว, คำอธิบายไฟล์ทีละไฟล์, สิ่งที่รู้กันอยู่แล้ว ("เขียนโค้ดสะอาด")

⚠️ การแก้ความเข้าใจผิด: ตัวเลข "~200 บรรทัด" เป็น soft guideline ไม่ใช่ hard threshold ที่วัดมา (refuted 0-3 ในการ verify). อย่าเล็งที่จำนวนบรรทัด — ใช้ per-line test. และ "ไฟล์ยาวเกิน" คือสาเหตุที่ถูกวินิจฉัยเวล Claude ไม่ทำตามกฎที่มีอยู่ → prune หรือเปลี่ยนเป็น hook อย่า restate.

ที่มา: anthropic.com/engineering/claude-code-best-practices, code.claude.com/docs/best-practices

C — Persona: slim ไม่ใช่ลบ

งานวิจัย (Zheng EMNLP-2024, Wharton, PRISM): persona แบบ verbose ไม่ช่วย ความแม่นเชิงข้อเท็จจริง (อาจทำให้แย่ลงด้วยซ้ำ). แต่ นั่นคือเรื่อง accuracy ไม่ใช่ tone/voice/role. เอกสาร Claude 4 บอก: "ตั้ง role แค่ประโยคเดียวก็ทำให้ต่างได้" (ด้าน tone).

⇒ slim identity ให้เหลือไม่กี่บรรทัดที่ signal สูง — อย่าขยายเพื่อหวัง accuracy, แต่ก็อย่าลบทิ้ง (มันช่วยจัด tone/voice). (หมายเหตุ: null-result นี้มีงานแย้ง 1 ชิ้นว่า baseline เองก็มี persona)

D — Orchestration: ตัด prose ทั่วไป เก็บ "ข้อเท็จจริง"

Frontier model "delegate ให้ subagent เองโดยไม่ต้องสั่ง" → prose ทั่วไปแบบ "delegate เมื่อเหมาะสม" ตัดได้. ความต้องการจริงคือคุมไม่ให้ over-delegate ไม่ใช่สั่งให้ delegate.

⚠️ version-fragile + เกี่ยวกับเราตรง ๆ: Opus 4.6 spawn เยอะเกิน; Opus 4.8 (runtime เรา) spawn น้อยลง → bias อาจกลับด้าน. คำถามเปิด: บน 4.8 เราอาจต้องเก็บ/กระตุ้น delegation มากกว่าจะตัด — ต้องดูจากพฤติกรรมจริง ไม่ใช่เดาจาก doc.

เก็บ: ข้อเท็จจริงเฉพาะ GLUEBOY (BOY ไหนทำอะไร, รูปคำสั่ง maw เป๊ะ ๆ, codex --force/bypass gotcha) — เฉพาะโปรเจกต์ + เดาไม่ได้. ตัด: ปรัชญา delegation ทั่วไป.

ที่มา: docs.claude.com/.../claude-4-best-practices, claude-code issue 27645

E — Must-fire → hooks; emphasis ใช้ได้

กฎที่ต้องเกิดทุกครั้ง → PreToolUse hook (CLAUDE.md เป็น context ไม่ใช่ config บังคับ). กฎที่ hook คุมอยู่แล้ว เหลือ pointer 1 บรรทัดได้.

✅ "CRITICAL / You MUST" ไม่ใช่ anti-pattern (refuted 0-3) — emphasis ยังเป็น lever เพิ่ม adherence ที่ถูกต้อง. ตัวแก้ "กฎโดนเมิน" คือ prune ความยาว / ย้ายไป hook ไม่ใช่ทำให้คำสั่งอ่อนลง.

F — วิธี "101": minimal-set
