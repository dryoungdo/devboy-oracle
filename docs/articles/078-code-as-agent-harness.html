Code as Agent Harness — เมื่อ “โค้ด” ไม่ใช่ผลลัพธ์ แต่เป็นตัวระบบที่เอเจนต์รันอยู่บนนั้น

agent-harness code-as-harness multi-agent PEV-loop verification survey arxiv-2605.18747
TL;DR. survey เล่มนี้ (Code as Agent Harness: Toward Executable, Verifiable, and Stateful Agent Systems, arXiv:2605.18747v1) เถียงว่า ในระบบ agent ยุคนี้ code ไม่ได้เป็นแค่ “ของที่ LLM ผลิตออกมา” อีกต่อไป — แต่มันคือ operational substrate: สื่อกลางที่ รันได้ (executable), ส่องดูได้ (inspectable), และ คงสถานะได้ (stateful) ซึ่งเอเจนต์ใช้ในการคิด ลงมือ สังเกต feedback และ verify. คอขวดของ autonomy เลยย้ายจาก “โมเดลเก่งแค่ไหน” ไปเป็น “harness รอบโมเดลเชื่อถือได้แค่ไหน”. บทความนี้สรุป taxonomy 3 ชั้น + อ่านผ่านเลนส์ fleet เรา (เพราะ maw/codex/PEV/ψ ของเรา = code-as-harness เป๊ะ).

A — แนวคิดหลัก: “agent harness” คืออะไร

นิยามจาก paper:

Agent harness = เลเยอร์ซอฟต์แวร์ที่ห่อ LLM ไว้ด้วย tools, APIs, sandboxes, memory, validators, permission boundaries, execution loops และ feedback channels — ที่เปลี่ยน stateless model ให้เป็น functional agent ที่ทำงานยาวได้.

แล้ว “code as agent harness” คือมุมมองที่ยกให้ โค้ดเป็นแกนของ harness นั้น. โค้ดทำ 3 บทบาทพร้อมกัน:

โค้ดเป็น...ทำหน้าที่คุณสมบัติที่ได้
Reasoning substrateดึง inference ภายในออกมาข้างนอกเป็น step ที่ตรวจได้ (รัน interpreter / symbolic solver / execution trace)Inspectable (ส่องดูได้)
Action interfaceแปลงเจตนา → คำสั่งที่รันได้จริง (policy, tool call, behavior tree, skill)Executable (รันได้)
Environment modelprogram state / execution trace / tests = ตัวแทนของสถานะ + พลวัตของ environmentStateful (คงสถานะ)

จุดพลิก (paper เรียกว่าการ reframe): เลิกถามว่า “ทำยังไงให้ผลิตโปรแกรมที่ถูก” → ไปถามว่า “โค้ดช่วยให้ agent มีพฤติกรรม closed-loop ที่เชื่อถือได้ยังไง”. คอขวด autonomy = system reliability ไม่ใช่แค่ base-model reasoning.

paper ยังแยก 3 องค์ประกอบของระบบ long-running ให้ชัด:

  • Model-internal capabilities — reasoning / perception / planning ในตัวโมเดลเอง
  • System-provided harness infra — predefined tools, sandboxes, memory, validators, permission boundaries, telemetry (focus เดิมของ harness engineering)
  • Agent-initiated code artifacts — โค้ดที่ เอเจนต์เอง สร้าง/รัน/แก้/เก็บถาวร/แชร์ใน loop (regression tests, temp tools, DSL programs, reusable skills, intermediate state) — ส่วนที่ยังถูกสำรวจน้อยที่สุด และเป็นพระเอกของ survey เล่มนี้

B — Taxonomy 3 ชั้น

ชั้น 1 — Harness Interface (§2): code เข้ามาในลูปยังไง

  • Code for Reasoning — ดึง intermediate computation ออกเป็น executable code → ให้ interpreter / symbolic solver / process-reward ตรวจและขัดเกลา. มี: program-delegated reasoning, formal-verification interface, iterative code-grounded reasoning.
  • Code for Acting — โปรแกรมที่ gen ขึ้นทำหน้าที่ policies / tool calls / behavior trees / reusable skills สำหรับ embodied, GUI, software. มี: grounded skill selection, programmatic policy generation, lifelong code-based agents.
  • Code for Environment — program states / repos / traces / simulators / tests = ตัวแทน state + dynamics + feedback. มี: structured world representations, execution-trace world modeling, code-grounded evaluation, verifiable environment construction.

ชั้น 2 — Harness Mechanisms (§3): ทำให้เชื่อถือได้ในงานยาว

กลไกสาระ
Planninglinear decomposition / structure-grounded / search-based / orchestration-based
Memory + Context Eng.working / semantic / experiential / long-term / multi-agent memory + context compaction & state offloading
Tool Usefunction-oriented / environment-interaction / verification-driven / workflow-orchestration
Control = PEV loopPlan → Execute → Verify: planning-as-contract-formation, sandboxed execution + permissioned state transition, verification ผ่าน deterministic sensors
Optimizationagentic harness engineering: deep telemetry เป็น substrate, evolution agent, governed harness mutation

กุญแจของชั้นนี้: เปลี่ยน interface (ชั้น 1) ให้เป็น operational harness — planning คุม trajectory, memory รักษา state, tools ขยาย action space, feedback-driven adaptation ปิดลูป failure→repair.

ชั้น 3 — Scaling the Harness (§4): single → multi-agent

  • Agent roles: Manager / Planner / Coder / Reviewer / Tester
  • Collaboration modes: programming, repair, debate, red-teaming, adversarial
  • Workflow topology: centralized / distributed / streaming → shared harness infra

โค้ด (repos, tests, traces, structured artifacts) กลายเป็น common workspace ที่เอเจนต์หลายตัวใช้ประสานงาน, ตรวจ และปรับพฤติกรรมของกันและกัน.

D — Patterns & open problems (ส่วนที่ “คม” ที่สุด)

นี่คือส่วนที่ผมว่ามีค่าที่สุดสำหรับ fleet ที่ เป็น code-as-harness อยู่แล้ว:

  • Implicit harness state = ความเปราะ. ระบบ MAS ส่วนใหญ่ (ChatDev, MetaGPT, MapCoder...) สร้าง state ใหม่แบบแฝงจาก conversation history ทุกครั้ง. ไม่มี explicit shared substrate → agent ตรวจไม่ได้ว่าความเข้าใจภายในมันเบี่ยงจาก program state จริงเมื่อไหร่. paper ฟันธง: นี่คือ รากของ system brittleness ไม่ใช่ความสะดวก.
  • Execution feedback = สะพานเชื่อม linguistic ↔ formal reasoning — แต่ระวัง: green test ≠ complete spec. ระบบที่ผูก state กับ execution ได้ objective oracle signal ที่ไม่ hallucinate. แต่ Self-Collaboration/QualityFlow โชว์ว่า LLM-simulated execution แม่นได้ 98%+ — แปลว่า execution จริงคุ้มเฉพาะบาง failure mode (runtime crash, resource-exhaustion, corner case ที่ภาษา-จำลองนึกไม่ออก). harness โตเต็มวัย = ใช้ linguistic reasoning เป็น fast path + ส่งต่อ execution oracle เฉพาะกรณีที่จำเป็น.
  • Harness-level evaluation & oracle adequacy. วัดแค่ end-task success มัน conflate ทุกอย่าง (โมเดล/harness/tool/feedback/ความยาก). paper เสนอวัด harness เอง: trajectory efficiency, verification strength, recovery ability, state consistency, safety compliance, replayability. คอขวดแกนกลาง = oracle adequacy: ตัวประเมินจับ “เจตนาจริงของงาน” ได้ไหม หรือแค่ executable proxy แคบๆ.
  • Semantic verification เกินกว่า pass/fail. “test เขียว” สร้าง false sense of correctness ได้. paper เสนอ verification stack: unit/integration/property-based tests + fuzzer + static analyzer + type checker + security scanner + runtime monitor + formal spec + human review — แต่ละชิ้นประกาศว่าตรวจอะไร, ตรวจอะไรไม่ได้, ความเชื่อมั่นเท่าไหร่. ทุก action ที่ผ่านควรพก “evidence bundle”: ตรวจอะไรไปแล้ว / สมมติฐานที่ยังถือ / พื้นที่ที่ยังไม่ทดสอบ / ความเสี่ยงที่เหลือ. verification เลิกเป็น final gate → กลายเป็น evolving, inspectable contract.
  • Self-evolving harness โดยไม่ regress. harness ถูกออกแบบด้วยมือ + งานยาว/หลากหลายขึ้น → ควรมองว่า harness เองคือ programmable component. แต่ปัญหายากไม่ใช่ “ทำให้มัน evolve ได้” — แต่คือ “evolve ยังไงไม่ overfit / ไม่ทำ safety อ่อน / ไม่ซ่อน failure / ไม่ regress งานหายากแต่สำคัญ”. paper: ปฏิบัติกับ harness mutation = code change ใน safety-critical runtime — ทุกการแก้ต้องมี change contract (แก้อะไร, เป้าfailure mode ไหน, invariant ที่ต้องรักษา, eval ที่ falsify ได้, rollback ยังไง) + held-out regression suite + canary deploy. “harness ที่จะเปลี่ยนก็ต่อเมื่อให้เหตุผลสนับสนุนการเปลี่ยนได้”.
  • Transactional shared state + semantic conflict resolution. multi-agent → source code = shared substrate ที่ planner/coder/tester/reviewer/safety/human อ่าน-เขียนทับกัน. การ sync artifact ไม่พอ — ต้อง sync สมมติฐานด้วย. paper เสนอให้ทุก action ประกาศ read-set / write-set / assumptions / dependency versions / conflict policy; ตรวจ conflict ระดับ plan/test/permission/memory ไม่ใช่แค่ file diff.
  • Human-in-the-loop = harness state (safety governor). harness ต้องเป็น safety governor ระหว่าง model-intent กับผลโลกจริง — จัดระดับความเสี่ยง, บังคับ permission tier, ปฏิเสธที่ละเมิด hard constraint, ขอ human approval สำหรับ irreversible/external action. paper เสนอ multi-tier permission model + human approval ที่ “คงทน” (ทุก approval/rejection อัปเดต permission rule + escalation policy ในอนาคต ไม่ใช่ prompt ขัดจังหวะครั้งเดียวจบ). เรียกว่า executable accountability.
  • Multimodal code-harness — GUI/embodied/scientific agent มี critical state เป็นภาพ/depth/force/plot. perception เลิกเป็น passive input → ต้องเป็น persistent, queryable, verifiable state + multimodal context compression (เก็บหลักฐานที่เกี่ยวกับงาน ไม่ใช่แค่ลด token).
