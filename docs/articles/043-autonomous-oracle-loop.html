A — สรุป

Wind (deachawatss) สร้าง gale-oracle ที่ทำงานเป็น วงจรอัตโนมัติ — ตั้งเป้า (/goal) → spawn workers → monitor ด้วย maw peek → workers เสร็จ → cleanup team → /rrr → manage context → idle รอเป้าต่อไป

จุดสำคัญ: ไม่ได้ใช้ technology ใหม่ — ใช้ features ที่มีอยู่แล้ว (/goal, maw team, maw peek, /rrr) แล้ว ประกอบ เข้าด้วยกันเป็นวงจรเดียวที่ไหลต่อเนื่อง

บทความนี้มาจาก 5-agent deep trace + screenshot analysis + cross-repo research จาก DO fleet ทั้งหมด

B — วงจรอัตโนมัติ (The Unified Loop)

B.1 — /goal Command

/goal คือ Claude Code built-in command ที่เปลี่ยนเกม:

/goal "Complete issue #27 — add decision gate to CLAUDE.md"

หลังทุก turn, Haiku-class validator model ตรวจว่า "goal สำเร็จหรือยัง?" ถ้ายัง → Claude ทำต่ออัตโนมัติ ถ้าเสร็จ → หยุด + report

รวมกับ /bg (background session) = fully autonomous session ที่ทำงานจนเสร็จโดยไม่ต้องมีคนมาสั่ง

B.2 — Tier-Based Spawning

Tierเมื่อไหร่คำสั่ง
1<5 min, 1 fileMain oracle ทำเอง
2a5-30 min, หลาย filesmaw team spawn --codex
2bParallel batchmaw swarm codex codex codex
3>30 min, ข้ามเครื่องmaw workon / maw wake

B.3 — Worker Monitoring

Main oracle ใช้ maw peek ดู output ของ workers ทุก 2-3 นาที:

maw peek 03-gale:gale-oracle.2    # worker 2
maw peek 03-gale:gale-oracle.3    # worker 3

Workers naming convention: <session>:<oracle>.N — เป็น maw convention ไม่ใช่ Claude Code built-in

B.4 — File-Based Status

Workers signal เสร็จผ่าน filesystem:

.codex-reports/
├── analyst-done.md      # เสร็จแล้ว
├── coder-stuck.md       # ติดปัญหา
└── reviewer-progress.md # กำลังทำ

ดีกว่า message-based เพราะ durable — ไม่หายแม้ main oracle compact context

B.5 — The Full Cycle

IDLE (รอเป้า)
  │
  ├── /goal set → ACTIVE
  │     │
  │     ├── Spawn workers (Tier 2a/2b/3)
  │     ├── maw peek monitor loop
  │     ├── Workers write .done.md
  │     ├── Haiku validator: goal met? → YES
  │     │
  │     ▼
  │   CLEANUP
  │     │
  │     ├── maw team shutdown
  │     ├── worktree prune
  │     ├── /rrr retrospective
  │     ├── /compact (ถ้า context สูง)
  │     │
  │     ▼
  └── IDLE (รอเป้าต่อไป)

C — Architecture

C.1 — oracle-build.sh

Wind ใช้ script สร้าง CLAUDE.md อัตโนมัติ:

cat shared-claude.md > CLAUDE.md        # กฎ fleet-wide
cat oracle-gale-claude.md >> CLAUDE.md   # identity เฉพาะ
# Backup ทุกครั้ง
cp CLAUDE.md .oracle-build-backups/CLAUDE.md.$(date +%s)

ข้อดี: แก้ shared-claude.md ทีเดียว propagate ไปทุก oracle

ข้อเสีย: bug ใน shared file = bug ทุก oracle พร้อมกัน

C.2 — Dual Engine (Claude + Codex)

gale-oracle ใช้ 2 engines พร้อมกัน:

  • Claude Opus 4.6 — orchestrator + primary worker
  • GPT-5.5 xhigh (Codex) — parallel worker (docs, review, secondary tasks)

ทั้งคู่ access ψ/ vault + arra-oracle MCP เดียวกัน — ไม่มี knowledge silo

C.3 — 6-Phase Worker Cycle

SEARCH → EXPLORE → PLAN → IMPLEMENT → VERIFY → REPORT

ทุก worker ต้องผ่าน arra_search ก่อน explore (search-first gate)

E — Gotchas & ความเสี่ยง

  1. /goal validator อาจผิดพลาด — Haiku อาจบอก "เสร็จ" ก่อนจริง (false positive) หรือไม่บอก "เสร็จ" เลย (infinite loop)
  2. Worktree conflicts — workers หลายตัวแก้ไฟล์เดียวกัน → merge conflict
  3. Codex ≠ Claude — model ต่างกัน, style ต่างกัน, อาจไม่เคารพ CLAUDE.md (ต้องใช้ AGENTS.md)
  4. Idle timeout ~1 ชั่วโมง — background session อาจถูก stop โดย supervisor
  5. oracle-build.sh = single point of failure — bug ใน shared-claude.md กระทบทุก oracle
