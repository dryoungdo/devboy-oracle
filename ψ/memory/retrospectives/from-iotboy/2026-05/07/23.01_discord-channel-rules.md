## AI Diary

First I assumed the repo's `access.json` schema (`allowChannels`, `commandAuthority`) was the runtime schema — it wasn't, those were Captain's bookkeeping. Three sources disagreeing about what `access.json` should look like: the plugin's actual code, the skill's documentation, and the repo's existing config. I should have read the plugin source on the very first failure instead of trusting the skill docs. "When the error message and the docs both seem authoritative, read the source." That's the lesson. Saved it.

## Honest Feedback

**Friction 1 — three-source schema disagreement.** The repo's `access.json`, the skill's documentation, and the plugin's actual code all describe `access.json` differently. The repo had an `allowChannels` field that does nothing. The skill describes `groups` with `requireMention`/`allowFrom`. The plugin requires `groups` keyed by channel ID. None of them said "by the way, your existing file is half-fictional." Future sessions need a cleanup pass: either drop `allowChannels`/`commandAuthority`/`respectRoles` from access.json or document them as IOTBOY-specific metadata (which I did with underscore prefixes).

**Friction 2 — error message points at the wrong tool.** "channel X is not allowlisted — add via /discord:access" is misleading when `DISCORD_STATE_DIR` is set, because the skill writes to the default path. The plugin could check the override and adjust the hint, or the skill could detect the override and warn. I logged this as a project memory but didn't file it upstream — should I?

**Friction 3 — I edited access.json directly instead of through the skill.** The skill's whole point is to prevent ad-hoc edits. I bypassed it because it writes to the wrong location for this project. That feels wrong even though it worked. The right fix is `DISCORD_STATE_DIR=.../discord-state /discord:access group add ...` — but the skill doesn't accept that env var either. So I'm stuck with manual edits until either tool gets fixed.

## Lessons Learned

1. **Read the source on first failure.** When error messages and skill docs both seem authoritative but disagree, the plugin's actual code is the only truth.
2. **Env var overrides change the rules.** `DISCORD_STATE_DIR` silently redirects all skill→plugin assumptions. Check `start.sh`/env early when a tool's output doesn't match its docs.
3. **Underscore-prefix unknown fields.** Plugin parsers ignore them, so they're a safe place to keep self-documenting metadata alongside runtime config.
4. **Channel-ID snowflakes, not friendly names.** Anywhere a config touches Discord routing, assume the runtime expects the snowflake.
