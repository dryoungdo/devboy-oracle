## Honest Feedback

1. **Pordee's benchmarking infrastructure exists but wasn't run.** There's a `benchmarks/` directory with `run.js` and `prompts.json`, but no `compression.json` was ever generated. The savings calculation literally can't work without it. So pordee's own stats tool was showing "no benchmark available" — and nobody noticed for 10+ days of usage. The plugin ships with measurement tooling that was never exercised. This is a gap in the plugin's onboarding, but also in my own diligence.

2. **The measurement methodology has a confound.** I compared this session (pordee ON) to fleet sessions (pordee OFF), but the sessions differ in task type, duration, and model version. A proper A/B test would answer the same prompt with pordee ON then OFF. My analysis controls for this somewhat by looking at text% across similar tool-heavy sessions, but it's not a clean experiment. I should have noted this limitation more prominently in the report.

3. **Fleet token analytics don't exist.** I had to write ad-hoc Python scripts to parse .jsonl files and calculate token splits. The fleet has no standard tool for "show me output token breakdown by type." This should exist — `/dig` mines sessions for timeline, but nothing mines for token economics. The pordee validation exposed this tooling gap.

## Lessons Learned

1. **Vendor claims need measurement, not acceptance** — 60-75% overstated to 2-3% for real workload. The gap was 25x. Always run the numbers on Day 1.
2. **Text is <5% of Oracle output** — any optimization targeting text-only has a hard ceiling of ~5% total impact for tool-heavy sessions.
3. **Honor-system safety modes are not safety modes** — pordee's security auto-disable is a prompt instruction. Gate-layer (hook) enforcement is required for reliability.
