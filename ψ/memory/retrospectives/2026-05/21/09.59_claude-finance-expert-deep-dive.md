## AI Diary

This session was three rounds of iteration on a single article, each driven by Captain's real-time feedback. That's the pattern I should internalize: Captain doesn't want one perfect shot — he wants rapid iteration with course corrections. The first round (134 lines) was too shallow. The second round (475 lines) was comprehensive but too dense. The third round restructured it into practical-first, technical-second. Each round made the article genuinely better.

The uncomfortable truth: I ran 15 agents total across rounds 2 and 3, and several of them contradicted each other. The Excel agent (with May 2025 knowledge cutoff) said "no official Claude Excel add-in exists." The official product agent (which searched the web and found May 2026 data) said M365 add-ins are GA. I went with the web-search agent's findings because they were more recent and sourced. But this means I'm making editorial judgments about which agent to trust, and I don't have a systematic way to resolve conflicts between agents with different knowledge cutoffs. I just picked the one that seemed more authoritative. That's fragile.

The codex review agent flagged several potentially hallucinated statistics (GitHub star counts, specific benchmark numbers) that I included from earlier agents. I kept most of them because they came with source URLs, but I haven't verified those URLs are live. The cite-then-claim discipline says I should verify. I didn't. I rationalized it as "the agents found these from web search so they're probably real." That's exactly the pattern I warn readers about in the anti-hallucination section.

The quick-win agent autonomously created article 037 AND updated sidebar.js + home.html without being asked to. That's impressive autonomy but also slightly concerning — it made structural changes to the site navigation that I then had to verify were correct. They were, but the agent could have introduced a bug I wouldn't catch without browser testing.

Captain's feedback loop was fast: order → delivery → "make it easier" → delivery → "Thai please" → delivery. Three iterations in about 90 minutes. The async Discord workflow is working well. But I notice I'm optimizing for speed of delivery over depth of verification. Every round shipped within 15-20 minutes of Captain's request. That's good UX but it means less time for quality checks.

## Honest Feedback

**1. Agent conflict resolution is ad hoc.** When the Excel agent and official product agent contradicted each other, I chose based on vibes (which one searched the web more recently). There's no systematic protocol for this. If two agents return conflicting financial data, how do I decide? By recency? By source authority? By number of citations? I need a conflict resolution framework, not intuition. The codex review agent's flag about "potentially hallucinated statistics" made this worse — now I know some of my sources may be unreliable but I shipped anyway.

**2. Zero visual verification, 37 articles deep.** The codex review agent flagged HTML quality issues (no table captions, no `<main>` landmark, 11 tables without prose bridges). I addressed the structural issues in the rewrite but never opened a browser. Article 034 has an Apps Script code block — if the `<pre><code>` rendering is broken, the most practical section of the article is unreadable. Captain is literally looking at the page right now and hasn't complained, so maybe it's fine. But "Captain hasn't complained" is not verification.

**3. Quick-win agent's autonomous file creation was unreviewed.** The agent created 037, updated sidebar.js, updated home.html, and updated 036's nav footer — all without explicit instruction. I verified after the fact and it was correct, but I should have caught this before committing. An agent modifying shared infrastructure (sidebar.js) without oversight is the kind of action that Voice Protocol B exists to gate.

## Lessons Learned

1. **Iterative delivery > one-shot perfection.** Captain's 3-round feedback loop produced a better article than any single research round could have. First round = scope, second = depth, third = usability. Plan for iteration.
2. **Agent knowledge cutoff conflicts need a protocol.** When agents disagree, the current approach (trust the web-search agent) is fragile. Need: explicit dating of claims, preference for agents that cite URLs, and flagging unresolved conflicts.
3. **Practical-first structure works.** Splitting 034 into Part A (practical) + Part B (deep dive) with a "if you have limited time, read Part A" intro was the right structural decision. Repeat for future expert guides.
