## Honest Feedback (session-specific friction)

1. **arra-cli thread output is truncated and there's no `--full` / `--no-truncate`
   flag.** GLUEBOY's first dispatch ended mid-sentence at "Apple Silicon on"
   because arra-cli capped the message-body display in terminal mode. I had to
   query sqlite directly: `sqlite3 ~/.arra-oracle-v2/oracle.db "SELECT content
   FROM forum_messages WHERE thread_id=N"`. Acceptable workaround for now, but
   every long-dispatch read will hit this.
2. **ToolSearch keyword matching pulled noise.** Searching "arra thread read"
   returned MCP Gmail thread tools (because of the word "thread"). Took two
   searches before I confirmed there's no MCP arra layer here — only the CLI.
   Not a blocker; just a reminder that "thread" is overloaded across plugins.
3. **CLAUDE.md says GLUEBOY Quick-mode dispatch arrives via `maw talk-to
   devboy '...' --force`, but in practice it arrived as an arra thread post
   with a maw heads-up notification.** Threads-first works fine; the doc line

## Lessons Learned (generalizable)

1. **GLUEBOY Quick-mode dispatches arrive via arra threads, not maw direct.**
   The canonical inbox is thread `channel:devboy` (#1). Maw notifications are
   the bell; the thread is the letter. Always read the thread before assuming
   the preview is the whole message.
2. **arra-cli thread output truncates message bodies in terminal mode.** For
   full payload, query the DB directly:
   `sqlite3 ~/.arra-oracle-v2/oracle.db "SELECT content FROM forum_messages
   WHERE thread_id=N"`. Until the CLI grows a `--full` flag, this is the
   workaround.
3. **Public-issue filing on external repos is autonomous when (a) it is the
   dispatch's deliverable and (b) the target org is fleet-friendly** (P'Nat's
   Soul-Brews-Studio, Captain's own forks, etc.). The first such filing per
   repo can carry a one-time content-confirmation check with Captain to
   establish the relationship; subsequent filings should not re-ask for
   authority.
