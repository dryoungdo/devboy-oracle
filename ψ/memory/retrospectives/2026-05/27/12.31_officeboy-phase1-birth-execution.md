## Honest Feedback

**Friction 1: Codex tile failure was silent and undiagnosable.** All 3 tiles showed `zsh` as CMD when I checked, meaning codex had exited. But `maw peek` returned garbled character-by-character output that was unreadable. I couldn't determine whether codex failed to start, started and crashed, or completed but didn't commit. The tile observability gap — no structured exit code, no log file, no "tile finished with status X" notification — makes swarm debugging guesswork. This is the same class of friction as the 2026-05-24 codex hang (issue #46), just at a different stage.

**Friction 2: Context compaction lost the tile dispatch details.** The pre-compaction session dispatched 3 tiles with detailed briefs. Post-compaction, I had a summary saying "tiles dispatched to panes .1, .2, .3" but couldn't verify what briefs were actually sent or whether they were received. The summary preserved intent but not the diagnostic data I needed to understand the failure. For tile-heavy work, the ScheduleWakeup + peek pattern needs to happen BEFORE compaction, not after.

**Friction 3: Same-branch swarm is architecturally wrong for this task.** All 3 tiles working on `main` of the same repo meant the second tile to push would conflict with the first. I knew this at dispatch time (the summary mentions it) but dispatched anyway. The right approach for "create N config files in one repo" is either: (a) single agent, or (b) each tile in its own worktree branch that gets merged sequentially. Swarm-by-default is the right policy for independent file slices, but oracle infrastructure setup is a single concern with shared dependencies (oracle-build.sh must exist before CLAUDE.md can be generated).

## Lessons Learned

1. **Codex tiles on DO may not have `codex` installed or accessible.** The tiles exited silently. Need to verify `which codex` on DO before dispatching tile work. This might be a DO-specific gap — codex works on Mac Studio (glueboy's tiles succeed regularly).
2. **Copy-and-adapt infrastructure tasks are solo work, not swarm work.** They require reading source files, understanding structure, and adapting for a new context. That's sequential comprehension, not parallelizable labor. The right swarm target is N independent code changes in N files.
3. **OFFICEBOY's settings.json follows the same pattern as DEVBOY** — minimal on DO. Added permissions + hooks that devboy doesn't have yet, which may cause DEVBOY to look inconsistent. Consider syncing devboy's settings.json to match.
