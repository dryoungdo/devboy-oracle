## 3. Core Abstractions & Relationships

### Data Model (D1/Drizzle)

Located: **src/db/schema.ts:1-35**

```typescript
webhookHits
  - id (PK, auto-increment)
  - endpoint: string (e.g. "line", "github")
  - suffix?: string (e.g. "/github", "/custom-path")
  - received_at: ISO timestamp
  - response_ms: number
  - body_length: number
  - body: text (first 4096 chars, or truncated GitHub parsing result)
  - forward_status?: number (3xx-5xx from outbound fetch)
  - forward_ms?: number (latency to forward destination)
  - forward_error?: string (e.g. timeout, DNS fail)

forwardRules
  - endpoint (PK) → forward_url mapping
  - forward_url: string (destination URL)
  - enabled: boolean (toggle forwarding on/off)
  - persist: boolean (if false, don't store webhooks in DB)
  - created_at, updated_at: ISO timestamps

aliases
  - id (PK, auto-increment)
  - value (UNIQUE) — raw ID from webhook (LINE userId, groupId, etc.)
  - label: string — human-readable name ("Alice", "Team Brews", etc.)
  - created_at: ISO timestamp
```

### Service Layer

#### stats.ts (Recording & Metrics)

- **recordHit(db, hit)** → Stores webhook to webhookHits table, returns row id
- **purgeOldHits(db)** → DELETE WHERE received_at < 7 days ago
- **getStats(db)** → Aggregates: total_requests (count), avg_response_ms, oldest_hit_at, recent 50 hits, forward_rules, aliases, D1 usage (size, writes/day)

#### forward.ts (Outbound Relay)

- **getForwardRule(db, endpoint)** → SELECT from forwardRules by endpoint (or null)
- **executeForward(db, hitId, forwardUrl, body, originalHeaders)** → Async fetch to forwardUrl, record forward_status/forward_ms/forward_error on hit

#### github.ts (Event Parser)

- **parseGitHubEvent(event, payload)** → Converts GitHub webhook events (push, PR, issue, release, etc.) into human-readable summaries

#### auto-alias.ts (LINE Discovery)

- **extractIds(body)** → Parses LINE webhook JSON, extracts unknown groupIds and userIds
- **fetchGroupName(groupId, token)** → Calls LINE Messaging API /v2/bot/group/{id}/summary
- **fetchUserName(userId, groupId, token)** → Calls LINE Messaging API /v2/bot/profile/{id} or /group/{gid}/member/{uid}
- **autoAlias(db, body, lineToken)** → Background task (called in waitUntil) — extracts new IDs, resolves names, upserts into aliases table

#### mcp.ts (MCP Server)

12 tools exposed via stdio/HTTP transport:
- **webhook_stats** — Dashboard snapshot
- **webhook_hits** — Query by date/endpoint
- **list_forward_rules**, **set_forward_rule**, **delete_forward_rule**
- **list_aliases**, **set_alias**, **delete_alias**
- **purge_old_hits**
- **generate_webhook_url** — Signs HMAC token
- **line_groups** — Groups + member count from hits
- **line_digest** — Parses LINE messages into readable table

### Authentication & Authorization

Located: **src/worker.ts:44-60**

- **generateWebhookToken(id, secret)** → HMAC-SHA256(id, API_TOKEN) encoded as URL-safe base64
- **verifyWebhookToken(id, token, secret)** → Constant-time equality check
- **getCookie(request, name)** → Parses "api_token" from cookie header
- **checkAuth(request, env)** → Allows if (1) no API_TOKEN set in env, (2) Bearer header matches, or (3) cookie matches

Auth flow:
1. External sender calls `/w/{id}/{token}` — token is verified (no session needed)
2. Dashboard user logs in at `/auth/login` → sets HttpOnly cookie `api_token`
3. API calls check cookie or Bearer header for API_TOKEN
4. MCP server uses env vars: WEBHOOK_RELAY_TOKEN or API_TOKEN

## 5. Data Flow: Webhook Arrival to Dashboard Display

### Path 1: Raw Webhook Reception

```
Sender (LINE, GitHub, etc.)
  ↓
POST /w/:id/:token
  ↓ (src/worker.ts:84-111)
1. Verify token: verifyWebhookToken(id, token, API_TOKEN)
2. Record to DB: recordHit() → webhookHits.id
3. If forwardRules[endpoint].enabled: executeForward() (async, waitUntil)
4. If endpoint="line" & LINE_CHANNEL_ACCESS_TOKEN: autoAlias() (async, waitUntil)
5. Return { ok: true, endpoint, received_at, response_ms }
  ↓
Client callback
```

**Key file**: src/worker.ts:84-111 (raw POST), 114-145 (GitHub variant), 148-171 (suffix variant)

### Path 2: GitHub-Parsed Variant

```
Sender (GitHub)
  ↓
POST /w/:id/:token/github
  ↓ (src/worker.ts:114-145)
1. Verify token
2. Parse event: parseGitHubEvent(x-github-event header, JSON body)
3. Record hit with parsed body (first 4096 chars) and suffix="/github"
4. Forward if enabled
  ↓
Database stores human-readable summary (not raw JSON)
```

### Path 3: Forward & Status Update

```
recordHit() writes initial row (no forward_status yet)
  ↓
executeForward() (runs async in waitUntil)
  ↓ (src/forward.ts:16-56)
Fetch forwardUrl with timeout 10s
  ↓
Update same row: forward_status, forward_ms, forward_error
  ↓
Dashboard polls /api/stats or /api/hits, reads row with forward status
```

### Path 4: Dashboard Display

```
User logs in → checkAuth()
  ↓
Dashboard.tsx polls /api/stats every 5s
  ↓ (src/worker.ts:417-427)
getStats(DB) aggregates:
  - total_requests: count(*)
  - avg_response_ms: avg(response_ms)
  - recent: order by received_at DESC limit 50
  - forward_rules: select * from forwardRules
  - aliases: select * from aliases
  - d1 stats: db size, writes/day
  ↓
React StatsGrid renders KPIs, WebhookFeed renders recent hits with alias resolution
```

**Files involved**:
- Recording: src/stats.ts:10-13 (recordHit)
- Forwarding: src/forward.ts:16-56 (executeForward)
- Dashboard API: src/worker.ts:417-427 (/api/stats)
- Frontend: frontend/src/pages/Dashboard.tsx:69-80 (poll loop)

## 8. Authentication & Security

### HMAC Signing (Webhook URLs)

Located: **src/worker.ts:24-42**

```typescript
token = HMAC-SHA256(id, API_TOKEN) → base64url-encoded
URL: /w/{id}/{token}
```

- No Bearer header needed for senders (Discord-style)
- Constant-time verification prevents timing attacks
- Token is deterministic (same id → same token if API_TOKEN unchanged)

### Session Auth (Dashboard)

- **Login**: POST /auth/login → verify username:password vs API_TOKEN → set HttpOnly cookie
- **Logout**: GET /auth/logout → clear cookie
- **Check**: GET /api/me → check cookie or Bearer header
- **Access control**: All /api/* and /mcp endpoints require checkAuth()

### Environment Secrets

| Var | Where Set | Usage |
|-----|-----------|-------|
| `API_TOKEN` | .dev.vars (local) or `wrangler secret put` (prod) | HMAC signing, auth check |
| `LINE_CHANNEL_ACCESS_TOKEN` | .dev.vars or wrangler secret | Optional: auto-fetch LINE display names |

## 13. Terminology & Patterns

- **Endpoint**: Identifier for a webhook receiver (e.g., "line", "github", "custom")
- **Token**: HMAC-SHA256 signature for endpoint (signed URL: /w/{id}/{token})
- **Hit**: A single webhook request stored in webhookHits table
- **Forward Rule**: Endpoint → URL mapping that enables async forwarding
- **Alias**: Mapping from raw value (LINE userId) to human label ("Alice")
- **Suffix**: Optional path suffix (e.g., "/github" to trigger GitHub parsing)
- **waitUntil()**: Cloudflare Workers runtime API to run async tasks after response sent (forwarding, auto-aliasing)
- **D1**: Cloudflare's serverless SQLite; 5 GB limit, 100k writes/day

## 15. Failure Modes & Edge Cases

### Webhook Verification Fails
- **Action**: Return 401, do not store
- **Code**: src/worker.ts:85-88

### Forward Timeout (10s)
- **Action**: Record forward_status=0, forward_error=message
- **Code**: src/forward.ts:40

### D1 Write Quota Exceeded
- **Action**: Drizzle throws error, hit fails to store (no retry)
- **Mitigation**: POST /api/purge to delete old hits

### LINE API Unreachable (autoAlias)
- **Action**: Continue; auto-alias task silently fails (catch block)
- **Code**: src/auto-alias.ts:99-114

### GitHub Parsing Invalid JSON
- **Action**: Fall back to raw text truncation
- **Code**: src/worker.ts:124-129

### No Forwarding Rule for Endpoint
- **Action**: Still store hit, skip forward
- **Code**: src/worker.ts:101-103

## Summary

**webhook-relay** is a **stateless webhook receiver** built on **Cloudflare Workers** that provides:

1. **Signed URL ingestion** (/w/{id}/{token}) — HMAC prevents spoofing
2. **Multi-backend storage** (D1/SQLite) — webhooks, rules, aliases
3. **Async forwarding** (10s timeout, status tracking)
4. **Domain-specific parsing** (GitHub events → summaries, LINE events → digest)
5. **Human-readable aliasing** (LINE userId → "Alice")
6. **React dashboard** (real-time polling, CRUD for rules/aliases)
7. **MCP integration** (12 tools for Claude Code)

The system is **pull-not-push**: external services POST to the signed URL; the dashboard polls for updates. No webhooks or callbacks required for the UI. All state is in D1; worker is stateless.

Typical latency:
- Webhook ingestion: <100ms
- Forward delivery: 100-5000ms (depends on destination)
- Dashboard poll: <500ms (query time)
- MCP tool call: 100-2000ms (depends on query complexity)
